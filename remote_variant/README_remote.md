# Разнесённое (split) развёртывание на отдельных серверах

Опциональный режим: тяжёлые компоненты (Qdrant + vLLM) выносятся на отдельный
**бэкенд-сервер** с GPU, а **сервер приложения** (FastAPI + эмбеддинги + LightRAG)
работает отдельно и обращается к бэкенду по сети. Полезно, когда GPU-машина одна,
а приложений/окружений несколько, или нужно разделить нагрузку.

Это не отдельная кодовая база — то же приложение, просто `QDRANT_URL` и
`LLM_BASE_URL` указывают на другой сервер. Одиночное развёртывание (всё на одной
машине) по-прежнему работает как раньше.

```
┌─────────────────────────┐         сеть          ┌──────────────────────────┐
│   Сервер ПРИЛОЖЕНИЯ      │                       │     БЭКЕНД-сервер (GPU)   │
│  FastAPI + веб-панель    │  ── Qdrant :6333 ───► │  Qdrant (векторная БД)   │
│  эмбеддинги / реранк     │  ── vLLM   :8001 ───► │  vLLM (генерация, CUDA)  │
│  граф LightRAG           │                       │                          │
└─────────────────────────┘                       └──────────────────────────┘
```

Что где выполняется:
- **Генерация (LLM)** — на бэкенде (vLLM, GPU).
- **Эмбеддинги, реранк, граф LightRAG** — на сервере приложения (CPU или его GPU).
- **Хранилище векторов** — на бэкенде (Qdrant).

## Установка

### 1. Бэкенд-сервер (Linux + NVIDIA)

```bash
# APP_HOST — IP сервера приложения (чтобы открыть порты только ему)
sudo VLLM_MODEL=Qwen/Qwen2.5-32B-Instruct-AWQ APP_HOST=<ip-app> \
     bash remote_variant/setup_backend.sh
```

Ставит Docker + NVIDIA toolkit, поднимает Qdrant (6333) и vLLM (8001),
настраивает firewall. По завершении печатает адреса и готовую команду для шага 2.

### 2. Сервер приложения (Linux)

```bash
sudo BACKEND_HOST=<ip-бэкенда> BACKEND_MODEL=Qwen/Qwen2.5-32B-Instruct-AWQ \
     DEVICE=cpu ADMIN_TOKEN='пароль' bash remote_variant/setup_app.sh
```

Создаёт окружение, прописывает в `.env` адреса бэкенда, ставит зависимости
(torch под `DEVICE`: `cpu` или `cuda`), проверяет связь и регистрирует
systemd-сервис `rag-api`. Откройте `http://<ip-app>:<порт>`.

`BACKEND_MODEL` должен совпадать с моделью, которую отдаёт vLLM на бэкенде
(`VLLM_MODEL`).

## Замечания

- **Безопасность.** Qdrant и vLLM поднимаются без авторизации. Держите их в
  приватной сети/VPN и ограничивайте доступ по `APP_HOST` (firewall). Не
  выставляйте порты 6333/8001 в публичный интернет.
- **Эмбеддинги на сервере приложения.** При `DEVICE=cpu` они медленнее; если на
  app-сервере есть GPU — ставьте `DEVICE=cuda` (torch соберётся под CUDA).
- **Граф LightRAG** строится на сервере приложения (кнопка «Построить граф» в
  админке): извлечение сущностей идёт через удалённый vLLM, эмбеддинги — локально.
- **Смена модели.** Меняйте `VLLM_MODEL` на бэкенде (перезапуск контейнера) и
  синхронно `LLM_MODEL` на сервере приложения (в админке — поле «Модель LLM»).
- **Несколько app-серверов.** К одному бэкенду можно подключить несколько серверов
  приложения — просто запустите setup_app.sh на каждом с тем же `BACKEND_HOST`.

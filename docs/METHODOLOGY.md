# Документация по методологии работы

Документ объясняет, как и почему система устроена именно так: подход RAG, полный
конвейер обработки, выбор моделей, механика поиска и генерации, три режима работы
и расширения (структурные метаданные, умные фильтры, граф-RAG). Цель — чтобы
администратор понимал не только «какие кнопки», но и «почему так» и «как
настраивать под качество».

---

## 1. Почему RAG, а не дообучение

Система построена на **RAG** (Retrieval-Augmented Generation): модель не
дообучается на корпоративных данных, а на каждый вопрос получает релевантные
фрагменты документов как контекст и отвечает по ним.

Преимущества для нашей задачи:
- **Актуальность.** Изменили прайс — переиндексировали, ответы сразу новые. При
  дообучении пришлось бы переобучать модель.
- **Точность и проверяемость.** Ответ опирается на конкретные фрагменты, к каждому
  даётся ссылка на источник; легко проверить.
- **Контроль галлюцинаций.** Жёсткий промпт + порог релевантности заставляют
  отвечать только по контексту и честно говорить «не знаю».
- **Стоимость.** Не нужны GPU-часы на обучение и разметка данных.

Дообучение (fine-tuning) оправдано для смены стиля/формата ответов или узкой
доменной терминологии, но не для «знания» меняющихся фактов — это работа RAG.

---

## 2. Конвейер целиком

Две фазы: офлайн-индексация (редко, по расписанию) и онлайн-ответ (на каждый
вопрос).

```
ИНДЕКСАЦИЯ (офлайн):
  документы → загрузчики (по типам) → нарезка на чанки → метаданные →
  эмбеддинги (bge-m3) → запись в Qdrant (вектор + payload + индексы)

ОТВЕТ (онлайн):
  вопрос → [маршрутизация режима]
        → определение фильтров (rule-based / LLM)
        → плотный поиск в Qdrant (+ фильтр)
        → лексический пере-ранг BM25
        → кросс-энкодер реранк (bge-reranker) → отбор по порогу
        → сборка контекста → LLM (Ollama/vLLM) → потоковый ответ + источники
        → запись в журнал
```

---

## 3. Индексация

### 3.1. Загрузчики по типам (`loaders.py`)
Каждый формат обрабатывается осмысленно, а не «сырым текстом»:
- **PDF** — постранично (сохраняется номер страницы для цитирования).
- **DOCX** — абзацы + содержимое таблиц.
- **PPTX** — по слайдам, включая заметки докладчика (часто это конспект обучения).
- **XLSX/CSV (прайс-листы)** — построчно в формат `колонка: значение`, чтобы модель
  не путала цены между строками — критично для каталогов.
- **TXT/MD/HTML** — текст.
- **Аудио/видео (записи обучения)** — транскрибируются Whisper (`mlx` на Apple,
  `faster-whisper` на GPU) и индексируются как текст.

### 3.2. Нарезка на чанки
Текст режется на фрагменты ~`CHUNK_SIZE` символов (умолч. 900) с перекрытием
`CHUNK_OVERLAP` (150), по возможности по границам предложений/строк. Перекрытие
не теряет смысл на стыках. Меньший чанк = точнее попадание, но больше фрагментов;
больший = шире контекст, но «грязнее». 900/150 — сбалансированное значение.

### 3.3. Метаданные
К каждому чанку пишется payload: источник, страница, тип файла, категория
(`price`/`presentation`/`training`/`document`), дата, заголовок. Категория и дата
определяются правилами (`metadata.py`) по имени/типу файла. По этим полям в Qdrant
строятся индексы — для быстрой фильтрации.

### 3.4. Инкрементальность
Для файла считается хеш (mtime + размер). Неизменённые файлы пропускаются,
изменённые — переиндексируются (старые чанки удаляются). Поэтому регулярный
`ingest.py` дёшев.

---

## 4. Эмбеддинги: bge-m3

Для семантического поиска используется **BAAI/bge-m3** (размерность 1024):
многоязычная модель с сильным русским, что важно для смешанных RU/EN корпусов.
Работает на GPU (CUDA) или Apple (MPS). Та же модель применяется при индексации и
при запросе — иначе вектора несопоставимы.

---

## 5. Поиск и реранк

Качество ответа determines качеством отобранного контекста. Используется
трёхступенчатый отбор (`retriever.py`):

1. **Плотный поиск** в Qdrant по косинусной близости — берём `TOP_K_RETRIEVE`
   кандидатов (умолч. 20), при необходимости с фильтром по метаданным.
2. **Лексический пере-ранг BM25** внутри кандидатов — добавляет чувствительность к
   точным совпадениям (артикулы, названия, числа), которые семантика иногда
   «сглаживает». Это и есть гибридность.
3. **Кросс-энкодер реранк** (`BAAI/bge-reranker-v2-m3`) — переоценивает пары
   «вопрос-фрагмент» и выдаёт нормированную релевантность 0–1. Оставляем
   `TOP_K_RERANK` лучших (умолч. 6) выше порога `MIN_SCORE` (умолч. 0.30).

Реранк — главный рычаг качества: он убирает «похожие, но не те» фрагменты. Порог
`MIN_SCORE` управляет честным «не знаю»: если ничего не превысило порог, система не
галлюцинирует, а сообщает об отсутствии ответа.

---

## 6. Генерация ответа

Отобранные фрагменты собираются в блок КОНТЕКСТ, к нему добавляется системный
промпт (`prompts.py`) с правилами: отвечать только по контексту, не выдумывать
цены/условия, отвечать по-русски, приводить источники, при отсутствии данных —
честно об этом сказать. Генерацию выполняет локальная LLM через единый интерфейс
(`llm_backend.py`): **Ollama** на Apple или **vLLM** на GPU. Температура низкая
(0.1) — для фактологии. Ответ отдаётся потоково, в конце — список источников.

---

## 7. Выбор LLM по платформе

- **Apple Silicon:** `Qwen2.5-32B-Instruct` (Q4) через Ollama — баланс качества и
  скорости при 64–96 ГБ; при желании 72B. Хороший русский, Metal «из коробки».
- **GPU (vLLM):** модель под VRAM (14B/32B/72B AWQ), continuous batching и paged
  attention дают высокий throughput при многих одновременных пользователях.

Имя модели меняется в админке; на GPU смена модели контейнера — кнопкой «Применить
модель LLM».

---

## 8. Режимы работы и расширения

Система предлагает три преднастроенных режима — от быстрого базового до полного
hybrid+. Это реализация принципа «дешёвый вектор как основа, тяжёлые приёмы — только
там, где они нужны».

### 8.1. Базовый
Вектор + реранк + быстрый rule-based авто-фильтр категории по ключевым словам
вопроса. Без дополнительных вызовов LLM — минимальная задержка. Покрывает основную
массу точечных вопросов («сколько стоит X», «что входит в Y»).

### 8.2. С метаданными
Добавляет два слоя структурности:
- **LLM-метаданные при индексации** (`enrich.py`): каждый документ один раз
  прогоняется через LLM, который извлекает `product`, `topic`, `doc_type`. Эти поля
  индексируются в Qdrant. На GPU это дёшево.
- **Умные фильтры из вопроса** (`query_filters.py`): перед поиском вопрос
  разбирается LLM в условия (категория/продукт/период) и применяется как фильтр
  Qdrant. Если фильтр даёт мало результатов — автоматический откат без фильтра
  (защита от пустой выдачи). Это резко повышает точность на каталогах/прайсах, где
  важно «именно этот продукт за этот период».

### 8.3. Hybrid+ (граф)
Добавляет **лёгкий граф-RAG** (`graph_rag.py` на базе LightRAG). Вопросы-«сводки»
(сравнения, обзоры, связи между сущностями, «все/в целом/тенденции») маршрутизируются
в граф знаний; точечные по-прежнему идут через вектор+реранк. Граф строится на той
же инфраструктуре — vLLM для извлечения сущностей/связей и bge-m3 для эмбеддингов,
без отдельного Ollama.

**Когда граф полезен.** Вектор отлично отвечает, когда ответ лежит в одном-двух
фрагментах. Но «глобальные» вопросы требуют синтеза по многим документам — здесь
граф сущностей и их связей даёт более связные ответы. Платой является дорогая
индексация (LLM прогоняет каждый документ), поэтому граф включают точечно.

Маршрутизация определяется по ключевым словам вопроса (`is_global`); список легко
расширить. При недоступности LightRAG слой молча откатывается на вектор.

## 8a. Дообучение модели (fine-tuning) и его роль

Дополнительно к RAG систему можно перевести на **дообученную** модель. Это не
замена RAG, а другой инструмент: RAG отвечает за знание фактов, дообучение — за
поведение модели (стиль, тон, формулировки, доменную терминологию, формат
ответов). Лучший результат — дообученная модель **в связке с RAG**.

Метод — **QLoRA**: на основе проиндексированных документов текущая LLM генерирует
датасет пар «вопрос-ответ» (`build_dataset.py`), затем обучается LoRA-адаптер в
4-bit (`train_lora.py`). Адаптер подаётся в vLLM через `--enable-lora` (веса не
сливаются), что позволяет мгновенно включать и откатывать дообученную модель.

В веб-панели появляется переключатель «Модель генерации»: **Базовая** или
**Дообученная (LoRA)**. Настройка `USE_FINETUNED` направляет запросы на адаптер;
имя адаптера — `FINETUNED_MODEL` (по умолчанию `company-lora`).

**Когда применять.** Если ответы фактически верны (RAG работает), но хочется
единый фирменный стиль, устойчивые формулировки или лучшее понимание внутренней
терминологии — дообучение помогает. Если проблема в фактах/полноте — это зона RAG
(чанкинг, пороги, метаданные), дообучение её не решит. Важно: не пытайтесь
«зашить» в модель меняющиеся цены — это работа RAG.

## 9. Настройка качества

Практические рычаги (все — в админке):
- Слишком много «притянутых» ответов → повысить `MIN_SCORE`, уменьшить `TOP_K_RERANK`.
- Система часто говорит «не знаю», хотя данные есть → понизить `MIN_SCORE`,
  увеличить `TOP_K_RETRIEVE`/`TOP_K_RERANK`, проверить чанкинг.
- Путаница между похожими товарами/периодами → режим «С метаданными»
  (LLM_METADATA + SMART_FILTER).
- Нужны сводные/аналитические ответы → режим «Hybrid+».
- Ответы слишком «вольные» → снизить `TEMPERATURE`, ужесточить `SYSTEM_PROMPT`.
- Не тот тон/формат ответов → редактировать `SYSTEM_PROMPT`.

## 10. Оценка и сравнение

Для проверки гипотез есть инструменты:
- **Журнал и аналитика** в веб-панели — реальные вопросы, доля ответов, частые темы,
  каких документов не хватает.
- **`compare.py`** — прогон одинаковых вопросов через вектор-пайплайн и LightRAG
  рядом, для оценки, оправдывает ли граф свою стоимость на ваших данных.

Рекомендуемый подход: запустить базовый режим, неделю-две собирать реальные
вопросы, и по доле «сводных» запросов решать, включать ли hybrid+ и метаданные.

## 11. Ограничения и принципы

- Ответы ограничены содержимым документов: чего нет в папке — того система не знает
  (это сознательный выбор против галлюцинаций).
- Качество транскрипции аудио/видео зависит от модели Whisper и качества записи.
- LLM-извлечение метаданных и умные фильтры добавляют вызовы LLM (задержка/нагрузка)
  — поэтому они опциональны и вынесены в режимы.
- Граф-RAG требует осознанного построения графа и заметных ресурсов на индексацию.
- Приватность по умолчанию: всё локально, данные не уходят наружу; при внешнем
  доступе обязателен reverse-proxy с авторизацией и заданный `ADMIN_TOKEN`.

# Документация по использованию

Руководство по повседневной работе с системой: веб-панель для сотрудников и
администратора, режимы работы, справочник всех настроек, операции обслуживания,
HTTP API и утилиты командной строки.

---

## 1. Запуск и доступ

После установки сервис доступен по адресу `http://<ip-сервера>:8000`.

- На **GPU-сервере** API работает как systemd-сервис `rag-api` (автозапуск).
  Управление: `bash gpu_variant/manage.sh {status|logs|restart|stop|start}`.
- На **Apple** запуск вручную: `uvicorn app:app --host 0.0.0.0 --port 8000`
  (можно оформить в `launchd`).

Сотрудникам достаточно открыть адрес и пользоваться вкладкой **«Чат»**. Остальные
вкладки — для администратора/аналитики.

---

## 2. Веб-панель

Боковое меню содержит разделы (дашборд, схема и тесты, чат, журнал, аналитика,
аналитика+, система, бенчмарк, администратор). Внизу — переключатель темы
оформления (Тёмная / Cyberpunk 2077 / Лето), текущий режим работы и активная
модель.

### 2.1. Дашборд
Сводные счётчики: всего запросов, за сегодня, доля отвеченных, средняя
релевантность, средняя задержка, **среднее время поиска и генерации** (этапы
запроса), число чанков в базе; плюс последние запросы. Кнопка «Обновить»
перечитывает статистику.

### 2.2. Схема работы и тесты
Актуальная схема всей связки: источники (документы/сайты/загрузка) → ингест
(+LLM-метаданные) → Qdrant; вопрос → режим/движок → вектор+реранк **или** граф
(LightRAG) → LLM (Ollama/vLLM, базовая/LoRA) → ответ; плюс опциональные удалённые
хосты. Кнопка «Демо-прогон» подсвечивает узлы.

Ниже — **«Тестирование компонентов»**: кнопка «Запустить тесты» проверяет каждый
компонент (настройки, Qdrant, эмбеддер, реранкер, LLM-генерация, папка документов,
журнал, граф, дообучение), показывает результат по каждому (OK/FAIL + детали),
общий вердикт и лог. Граф и дообучение считаются опциональными и не влияют на
общий результат. Эндпоинт: `POST /api/admin/selftest` (под токеном).

### 2.3. Чат
Основной интерфейс. Сотрудник задаёт вопрос — ответ приходит потоково, со
ссылками на источники (файл и, где есть, страница/слайд). Если в документах нет
ответа, система честно сообщает об этом, а не выдумывает.

**Подробности ответа.** Галка «Подробно об источниках и алгоритме формирования
ответа» под полем ввода добавляет к каждому ответу сворачиваемый блок: движок и
режим, модель и бэкенд, тайминги (поиск/генерация/всего), параметры поиска и
список использованных фрагментов (источник, страница, score, отрывок текста).
Состояние галки запоминается в браузере.

**Оценка ответа.** Под каждым ответом — кнопки 👍 / 👎. Оценки копятся в журнале,
видны в «Аналитике» (доля «хорошо», удовлетворённость) и используются для
калибровки настроек (см. раздел «Калибровка по оценкам» в админке).

**Визуальные источники.** Кроме текстовых ссылок, в блоке «Источники» показываются
сами артефакты: миниатюры изображений и RAW-фото, превью чертежей DXF/DWG (рендер),
плеер аудио, кадр-превью видео с кнопкой ▶ (открывает видео с перемоткой на нужный
момент) и ссылкой «⤓ фрагмент» (вырезает отрезок видео по таймкодам сегмента).
Превью строятся «лениво» при первом обращении и кешируются в `previews/` —
переиндексация не нужна; тайминги видео берутся из транскрипции (после переиндексации
аудио/видео в их чанках появляются `t_start`/`t_end`). Эндпоинты:
`GET /api/media/file|thumb|clip|info?source=…`. Для чертежей нужен `matplotlib`
(рендер ezdxf), для кадров/фрагментов — `ffmpeg`.

**Голосовой ввод и ответ.** Кнопка 🎤 у поля ввода записывает речь и распознаёт её
**локально** через тот же Whisper, что и для аудио/видео (данные не покидают сервер;
эндпоинт `POST /api/transcribe`): нажмите 🎤, говорите, нажмите ещё раз — текст
подставится в поле. Галка «🔊 Озвучивать ответы голосом» включает озвучку ответов
(синтез речи браузера, голос ОС, русский); у каждого ответа есть кнопка 🔊 для
ручного воспроизведения/остановки. Важно: доступ к микрофону браузеры дают только в
защищённом контексте — по `https://` или с `localhost`; при открытии по `http://<ip>`
запись будет недоступна (озвучка ответов работает и по http). Для голосового ввода в
сети поднимите HTTPS (reverse-proxy) или открывайте с самого сервера.

**Сессии и история.** Каждая сессия — отдельный чат со своим `session_id` (виден
как «чат #…» в шапке); все запросы помечаются им в журнале. Кнопка **«🆕 Новый
чат»** очищает окно и начинает новый чат (прежний уже сохранён в журнале). Кнопка
**«💾 Сохранить историю»** выгружает переписку текущего чата в Markdown-файл
(вопросы, ответы и источники). История одного чата также доступна по API:
`GET /api/chat-history?session_id=…`. Очистить историю всех чатов и статистику
можно в разделе «Администратор» (кнопка «🗑 Очистить историю всех чатов и
статистику», эндпоинт `POST /api/admin/clear-history`).

**Приложить документ к вопросу.** Кнопка 📎 рядом с полем ввода позволяет
«подложить» файл (Excel и др.) прямо к конкретному вопросу: файл парсится на лету,
самые релевантные строки подаются в контекст LLM — без добавления в общую базу и
без переиндексации. Удобно для разовых вопросов по конкретной таблице. Если файл
нужен постоянно — загрузите его в админке и переиндексируйте.

### 2.4. Журнал
Таблица всех обращений: время, вопрос, определённая категория, число найденных
фрагментов, релевантность, задержка, статус (ответ/нет данных) и источники.

### 2.5. Аналитика
Графики (Chart.js): запросы по дням, распределение по категориям, топ
цитируемых источников, распределение задержек, облако частых слов из вопросов.
Помогает понять, что спрашивают чаще всего и каких документов не хватает.

### 2.6. Система
Подробная информация и статистика по компонентам:
- **Qdrant** — число чанков, сегментов, размерность вектора и метрика, статус
  коллекции, поля payload, разбивка чанков по категориям, сколько чанков получили
  LLM-метаданные.
- **LightRAG (граф)** — установлен ли пакет, построен ли граф, число сущностей,
  связей, чанков и документов, размер хранилища, текущий движок и режим графа.
- **Дообучение (LoRA)** — установлены ли зависимости, готов ли адаптер и активен
  ли он, число пар в датасете, размер адаптера, базовая модель.
- **Hybrid+ и движки** — текущий режим и состояние флагов (метаданные, умные
  фильтры, граф, авто-фильтр), а также сколько ответов дано каждым движком
  (векторный / граф / LightRAG).
- **Время обработки (индексация)** — длительность последней индексации, число
  обработанных файлов и чанков, среднее время на файл, суммарное время по всем
  файлам и когда обновлялось (по данным `ingest_stats.json`).
- **Базы данных (SQLite / MySQL / PostgreSQL)** — по каждому бэкенду: статус
  (доступна / не настроена / нет драйвера / недоступна), какой активен, версия
  сервера, размер БД, число записей (запросы web, запросы Телеграм, пользователи
  бота, ключи настроек в `kv_store`), адрес/файл. Показывает данные и в неактивных,
  но доступных БД (например, после копирования).
- **Кэш Redis** — статус и расширенная статистика: версия, занятая память, число
  ключей кэша приложения и всего ключей в БД Redis, число клиентов, аптайм, процент
  попаданий кэша (hit rate), hits/misses, вытеснённые ключи. Если выключен или
  недоступен — об этом сообщается.
- **Каталог документов** — расширенная информация по файлам папки знаний: сводка
  (всего файлов, проиндексировано/нет, объём, разбивка по типам) и таблица с
  поиском по имени: файл, тип, размер, число чанков, статус (в базе/нет) и дата
  изменения. Есть **сортировка** (по имени, дате, размеру или числу чанков —
  кликом по заголовку столбца или выпадающим списком, с переключением направления)
  и **пагинация** (выбор размера страницы 50/100/200/500 и кнопки «Назад/Вперёд»).
  **Файлы с ошибками** подсвечиваются: после запуска «Проверить каталог с данными»
  в каталоге появляется счётчик «С ошибками», у проблемных строк — пометка
  «⚠ ошибка» с описанием причины (пустой/нечитаемый файл и т. п.), есть фильтр
  «только с ошибками» и сортировка «ошибки сверху» (клик по заголовку «Статус»).
  **Время обработки**: в сводке — карточка «Общее время обработки» и строка о
  последней индексации (длительность, число файлов, среднее на файл); в таблице —
  столбец «Время обр.» по каждому файлу с сортировкой (клик по заголовку). Данные
  берутся из `ingest_stats.json`, который пишет `ingest.py` при индексации.
  **Способ извлечения текста**: каждый файл помечается — 🎤 транскрибация
  (аудио/видео → Whisper), 🔍 OCR (изображения/RAW-фото) или 🛠 спец-инструмент
  (DWG/STEP/IGES, `.doc`, `.msg`, архивы). В сводке — счётчики «Транскрибировано» и
  «Распознано»; фильтр «Способ» позволяет отобрать транскрибированные/распознанные/
  и т. д. Кнопка 👁 в строке **раскрывает превью и содержимое файла**: миниатюра
  изображения/RAW-фото, рендер чертежа DXF/DWG, первая страница PDF, кадр видео с
  кнопкой ▶ (открывает плеер), проигрыватель аудио; ниже — извлечённый текст
  (транскрипция/распознанное) для проиндексированных файлов и ссылка «скачать
  оригинал». Превью доступно и для ещё не проиндексированных визуальных файлов
  (`GET /api/media/thumb|file|clip`, `GET /api/admin/file-text`).
  Эндпоинт каталога: `GET /api/admin/files-catalog?limit=&offset=&sort=name|date|size|chunks|proc|error&order=asc|desc&q=&only_errors=&method=transcribed|recognized|ocr|tool|text`.
- **Проверка каталога данных** — кнопка «Проверить каталог с данными» сканирует
  всю папку и выдаёт: сводку (ок/медиа/пустые/неподдерживаемые/таймаут/ошибки),
  список неподдерживаемых типов с рекомендациями (конвертация, OCR, распаковка,
  установка `xlrd` и т.п.) и таблицу проблемных файлов (файл → проблема → решение).
  Разбор файлов идёт **параллельно** в пуле процессов (до 8, по числу ядер) — это
  кратно быстрее на больших каталогах; узкое место — нативный парсинг (PDF/Office),
  поэтому ускорение даёт именно многопроцессность, а не смена языка. Лимит времени
  на файл берётся из `FILE_PARSE_TIMEOUT` (превышение → статус «таймаут», файл не
  блокирует проверку). Медиа/архивы/OCR при проверке не разбираются.
  Эндпоинт: `POST /api/admin/check-data`.

### 2.6a. Загрузка сервера
Отдельный раздел с текущей загрузкой хоста в реальном времени (галка
«автообновление» опрашивает каждые 2 с). Источник метрик — `psutil` (плюс
`nvidia-smi` для GPU). Показывает:
- **CPU** — общая загрузка и по каждому ядру, число потоков/ядер, частота,
  load average;
- **Память и swap** — занято/всего/свободно с индикатором заполнения;
- **Диски** — таблица по всем разделам (точка монтирования, ФС, занято/всего/
  свободно, процент);
- **GPU** — для NVIDIA: загрузка ядра и видеопамяти, использовано/всего МБ,
  температура, питание (Вт), вентилятор; для Apple Silicon — встроенный GPU с
  единой памятью (живую загрузку без root не получить); если GPU нет — отметка,
  что вычисления на CPU;
- **Сеть и аптайм**, а также **топ процессов по памяти** (PID, имя, RSS, CPU%).

Эндпоинт: `GET /api/server-load`. Если `psutil` не установлен — отдаётся
сокращённый набор (CPU-ядра, load average, память из `/proc`, корневой диск).

### 2.6b. Телеграм-бот
Доступ к базе знаний через Telegram. Бот работает на том же RAG-конвейере
(поиск + реранк + LLM) и отвечает со ссылками на источники.

**Подключение.** Создайте бота у **@BotFather** в Telegram, получите токен и
вставьте его в «Администратор» → группа **«Телеграм-бот»** (`TELEGRAM_BOT_TOKEN`,
секретное поле). Нажмите «Сохранить настройки», затем в разделе **«Телеграм»** —
**«Перезапустить бота»**. Бот стартует автоматически при запуске сервиса, если
токен задан.

**Подтверждение пользователей.** По умолчанию новый пользователь, написавший боту,
попадает в **«Запросы на доступ»** и получает сообщение об ожидании. Администратор
в разделе «Телеграм» нажимает **«Подтвердить»** (бот уведомит пользователя) или
**«Отклонить»** (блокировка). Подтверждённые могут задавать вопросы; список всех
пользователей (подтверждён/заблокирован/ожидает) — там же, с действиями
заблокировать/разблокировать. Галка **«авто-подтверждение»** (`TELEGRAM_AUTO_APPROVE`)
открывает доступ всем без ручного подтверждения.

**Прокси.** Если Telegram заблокирован в вашей сети, в группе «Телеграм-бот» задайте
`TELEGRAM_PROXY` — бот будет ходить к `api.telegram.org` через прокси. Поддерживаются
`socks5://`, `socks5h://`, `http://`, `https://` (можно с `user:pass@`). **Важно:**
бот работает по HTTP Bot API, поэтому **MTProto-прокси (tg://proxy-ссылки) не
подходят** — они для клиентов Telegram; укажите обычный SOCKS5/HTTP-прокси. Текущий
прокси виден в карточке статуса бота. После изменения — «Перезапустить бота».

**Отдельная история.** Запросы и ответы из Телеграм хранятся **отдельно** от
веб-чата (таблицы `tg_users`/`tg_requests`) и показываются в блоке «История и
запросы из Телеграм» со своей статистикой (всего/сегодня/пользователей/доля
ответов/задержка) и кнопкой очистки. Эндпоинты: `GET/POST /api/admin/telegram/*`
(status, restart, users, approve, block, unblock, requests, clear-history).

### 2.7. Аналитика+
Расширенная аналитика по компонентам с графиками (Chart.js). В каждом разделе, кроме
графиков, есть **прогресс-бары** с наглядными долями: Qdrant — покрытие чанков
LLM-метаданными (product/topic/doc_type) и распределение по категориям; Граф —
величины сущностей/связей/чанков/документов относительно максимума и охват базы
графом; Дообучение — готовность по этапам (датасет · параметры · адаптер, зелёный
при 3/3) и наполнение датасета; Тайминги — доля поиска и генерации в общем времени
ответа и бенчмарк компонентов относительно самого медленного. Ниже — графики:
- **Qdrant** — чанков по категориям (кольцевая), по типам файлов (столбцы),
  покрытие LLM-метаданными (product/topic/doc_type) и распределение ответов по
  движкам.
- **Граф (LightRAG)** — сущности, связи, чанки, документы (столбцы) и сводные
  показатели (связей на сущность, размер хранилища).
- **Дообучение (LoRA)** — статистика датасета (число пар, средние длины вопроса
  и ответа), параметры LoRA (r, alpha, dropout, число целевых модулей), размер
  адаптера и гистограмма длины ответов.
- **Тайминги по компонентам** — среднее время этапов запроса (поиск/генерация/
  всего), **время обработки (индексация)** — длительность последнего прогона,
  число файлов, среднее на файл и суммарное время, результаты последнего бенчмарка
  по компонентам (мс) и длительность последних операций (индексация, граф,
  дообучение, парсинг сайтов).

### 2.7a. Бенчмарк
Отдельный раздел замера производительности. Кнопка «Запустить бенчмарк» измеряет
скорость каждого компонента и выводит результаты (мс) графиком и карточками + лог:
- **Эмбеддер** — мс/текст и текст/с на пачке;
- **Реранкер** — мс/пара;
- **Поиск + реранк** — среднее время полного retrieval по нескольким запросам;
- **Qdrant** — латентность REST-запроса;
- **LLM генерация** — время и **токены/с** (для Ollama берётся `eval_count`/
  `eval_duration`, для vLLM — `completion_tokens` и время).
Эндпоинт: `POST /api/admin/benchmark`; результаты — в `/api/admin/status`.

### 2.8. Администратор
Управление системой: переключатель режима и движка, операции обслуживания и полный
набор настраиваемых параметров. Защищён токеном (`ADMIN_TOKEN`), если он задан.

## 3. Режимы работы

Вверху раздела «Администратор» — панель «Режим работы». Клик по карточке мгновенно
применяет пресет (нужно ввести токен администратора в поле ниже, если он задан):

| Режим              | Что включает                                                      | Когда использовать |
|--------------------|------------------------------------------------------------------|--------------------|
| **Базовый**        | Вектор + реранк, rule-based авто-фильтр                           | Максимальная скорость, типовые вопросы |
| **С метаданными**  | + умные фильтры из вопроса (категория/продукт/период) через LLM   | Точность на прайсах/каталогах |
| **Hybrid+ (граф)** | + граф-RAG для сводных вопросов (нужен LightRAG)                  | Есть аналитические/сводные вопросы |

Если вручную изменить отдельные флаги, режим отображается как **«Свой»**. Текущий
режим виден в боковом меню на всех вкладках.

### Смена модели LLM

Панель «Модель LLM» в админке показывает текущий бэкенд и список доступных
моделей. Клик по модели переключает генерацию на неё (поле `LLM_MODEL`,
применяется сразу). Для **Ollama** (Apple) можно скачать новую модель: введите
имя (например `qwen2.5:14b-instruct`) и нажмите «Скачать модель» — загрузка идёт
в фоне. В блоке «Состояние и операции» отображается **живой прогресс**: текущий
этап, проценты, скачано/всего и скорость (прогресс берётся из HTTP API Ollama
`/api/pull`; при недоступности — фолбэк на CLI).

Для **vLLM** (GPU) у поля `VLLM_MODEL` в настройках есть **выпадающий список
моделей** и кнопка **«🔄 Обновить список»**: список объединяет курируемый каталог
vLLM-моделей и модели, которые сервер vLLM обслуживает прямо сейчас (запрос к
OpenAI-совместимому `/v1/models`). Выбор подставляет имя в `VLLM_MODEL`; затем
«Сохранить настройки» и «Применить модель LLM (рестарт vLLM)». Эндпоинт:
`GET /api/admin/vllm-models`.

### Движок ответов (векторный / LightRAG)

Над переключателем модели генерации — выбор «Движок ответов»:

- **Векторный (первый вариант)** — поиск + реранк по документам со ссылками на
  источники (рекомендуется по умолчанию).
- **LightRAG (граф)** — ВСЕ вопросы направляются в граф знаний. Требуется
  построенный граф — его можно собрать прямо из админки кнопкой **«Построить граф
  знаний (LightRAG)»** (ставит LightRAG и прогоняет документы через LLM; идёт в
  фоне, статус — в «Состояние и операции»). Если граф недоступен — автоматический
  откат на векторный движок.

Отличие от режима «Hybrid+»: там в граф идут только сводные вопросы, а точечные —
через вектор; переключатель движка же отправляет в LightRAG всё. Управляется
настройкой `ENGINE`.

### Модель генерации (базовая / дообученная)

Ниже панели режима — переключатель «Модель генерации» с двумя карточками:

- **Базовая модель** — исходная LLM (RAG поверх неё).
- **Дообученная (LoRA)** — модель, дообученная на ваших документах (адаптер в vLLM).

Клик переключает `USE_FINETUNED` мгновенно. Здесь же — **выбор базовой модели для
дообучения**: выпадающий список fp16-моделей, пригодных для QLoRA (Qwen2.5/Qwen3,
Llama 3, Gemma, Mistral, Phi-4), и кнопка **«🔄 Обновить список моделей»**. По
умолчанию база берётся автоматически из `VLLM_MODEL` (с отбрасыванием суффиксов
квантизации `-AWQ/-GPTQ/-Int4`); выбор в списке задаёт её явно (поле
`FINETUNE_BASE`), пункт «из VLLM_MODEL (авто)» возвращает автоопределение.
Эндпоинт: `GET /api/admin/finetune-models`.

Рядом — кнопки **«Запустить дообучение»** (собирает датасет и обучает LoRA-адаптер
на выбранной базе; долго, в фоне) и **«Применить дообученную модель»**
(перезапускает vLLM с адаптером). Карточка «Дообученная» доступна после того, как
адаптер обучен и применён. Дообучение — дополнение к RAG (стиль/терминология),
факты по-прежнему берутся из документов. Подробнее — `finetune/README_finetune.md`.

## 4. Справочник настроек

Все поля редактируются в админке и сгруппированы; у каждого — бейдж, как
применяется изменение: **на лету**, **после переиндексации**, **после рестарта**,
**рестарт vLLM**. Изменения сохраняются в `runtime_config.json`.

### Поиск и генерация (на лету)
- **TOP_K_RETRIEVE** (5–50, умолч. 20) — сколько кандидатов достаём из Qdrant.
- **TOP_K_RERANK** (1–15, умолч. 6) — сколько фрагментов остаётся после реранка и
  уходит в контекст LLM.
- **MIN_SCORE** (0–1, умолч. 0.30) — порог релевантности после реранка; ниже —
  ответ «не знаю». Выше порог = строже, меньше «притянутых» ответов.
- **TEMPERATURE** (0–1, умолч. 0.1) — креативность генерации; для фактов держите низкой.
- **AUTO_FILTER** (умолч. вкл) — быстрый rule-based фильтр категории по словам вопроса.
- **AUTO_CALIBRATE** (умолч. выкл) — авто-подстройка параметров по оценкам 👍/👎.
- **SYSTEM_PROMPT** — поведение ассистента и правила цитирования.

> В админ-панели под каждым параметром этой группы показано **подробное описание и
> рекомендации по выбору значения** (когда увеличить/уменьшить и на что это влияет).

### Генерация (LLM)
- **LLM_BACKEND** — `ollama` (Apple) или `openai` (vLLM). *После рестарта.*
- **LLM_MODEL** — имя модели для запросов.
- **OLLAMA_URL / LLM_BASE_URL / LLM_API_KEY** — адреса бэкендов.

> Под заголовком этой группы в админке показан блок с описанием бэкендов и
> рекомендациями по выбору модели генерации по объёму памяти и задаче. Полный
> справочник — [docs/MODELS.md](MODELS.md).

### vLLM (GPU)
- **VLLM_MODEL / VLLM_MAX_LEN / VLLM_TP** — модель контейнера, длина контекста,
  число GPU. *Применяются кнопкой «Применить модель LLM».*

> Под заголовком группы — описание моделей vLLM и рекомендации по VRAM (24/48/80 ГБ),
> плюс пояснения к `VLLM_MAX_LEN` и `VLLM_TP`. Полный справочник — [docs/MODELS.md](MODELS.md).

### Эмбеддинги и устройство
- **EMBED_MODEL / RERANK_MODEL** — модели поиска и реранка (после рестарта).
- **EMBED_DIM** — размерность эмбеддингов (после переиндексации). Меняйте вместе с
  EMBED_MODEL: bge-m3, multilingual-e5-large, Qwen3-Embedding-0.6B, jina-v3 — все 1024.
  После смены: «Сбросить индекс» → «Переиндексировать».
- **DEVICE** — `cuda` / `mps` / `cpu`.

> В админ-панели под каждым параметром этой группы выводится **подробное описание и
> рекомендации**: какие модели эмбеддера/реранкера выбирать, какую ставить EMBED_DIM
> и когда нужны «Сбросить индекс»/«Переиндексировать»/«Перезапустить сервис». Полный
> справочник моделей — [docs/MODELS.md](MODELS.md).

### Хранилище
- **QDRANT_URL / QDRANT_COLLECTION** — адрес и коллекция (после рестарта).
- **QDRANT_TIMEOUT** — таймаут запросов из чата, с (после рестарта).
- **QDRANT_INGEST_TIMEOUT** — таймаут операций индексации, с (применяется при
  следующей переиндексации). Повышайте, если на больших файлах бывает «timed out».

### База данных и кэш
- **DB_BACKEND** (`sqlite` по умолчанию | `mysql` | `postgresql`) — где хранятся журнал
  запросов, история Телеграм и аналитика. SQLite — локальный файл, без внешних
  сервисов. MySQL/PostgreSQL — для нескольких инстансов или централизованного
  хранения. Менять напрямую для переезда не нужно — есть кнопки «Мигрировать»
  (см. раздел 5).
- **MYSQL_HOST / MYSQL_PORT / MYSQL_USER / MYSQL_PASSWORD / MYSQL_DB** — подключение к
  MySQL/MariaDB. Базу создайте заранее (`CREATE DATABASE rag CHARACTER SET utf8mb4;`),
  таблицы приложение создаст само. Нужен пакет `PyMySQL`.
- **PG_HOST / PG_PORT / PG_USER / PG_PASSWORD / PG_DB** — подключение к PostgreSQL.
  Базу создайте заранее (`CREATE DATABASE rag;`). Нужен пакет `psycopg2`.
- **REDIS_ENABLED** (по умолчанию выкл.) + **REDIS_HOST / REDIS_PORT / REDIS_DB /
  REDIS_PASSWORD** — опциональный кэш тяжёлых агрегатов (статистика, аналитика,
  статистика Телеграм). Кэш сбрасывается автоматически при каждом новом запросе, так
  что данные не устаревают. Без Redis всё работает напрямую. Нужен пакет `redis`.

### Документы и индексация (после переиндексации)
- **DOCS_DIR** — папка с документами. Рядом с полем — кнопка **«Обзор…»**: открывает
  браузер папок на сервере (навигация по подпапкам и вверх, счётчик документов в
  папке). Выбрав папку, нажмите «Сохранить настройки», затем «Переиндексировать».
- **CHUNK_SIZE** (умолч. 900) / **CHUNK_OVERLAP** (умолч. 150) — нарезка текста.
- **OCR_IMAGES / OCR_RAW / PARSE_CAD / TRANSCRIBE_AV** — включать ли тяжёлые
  экстракторы: OCR картинок, OCR RAW-фото, чтение чертежей CAD, транскрибацию
  аудио/видео. Это самые медленные этапы. Для **ускорения индексации** большого
  каталога отключите их, сделайте быструю первую переиндексацию (PDF/DOCX/таблицы/
  текст проиндексируются за минуты), затем включите нужные и переиндексируйте ещё
  раз — благодаря инкрементальности добавятся только новые типы.
- **FILE_PARSE_TIMEOUT** (умолч. 0) — лимит времени на один файл, с; превышение —
  файл пропускается (защита от «зависания» на тяжёлом DWG/видео). Работает на
  Linux/macOS. Пропущенные по таймауту видны в логе и сводке индексации.

> Скорость индексации: при `21 с/файл` и тысячах файлов узкие места — это OCR
> картинок (~секунды на файл), конвертация DWG и транскрибация видео (минуты на
> файл). Отключение этих экстракторов на первый проход сокращает время в разы.
> Предупреждение openpyxl «Data Validation extension is not supported» —
> безвредное, теперь подавляется.

### Транскрибация (после переиндексации)
- **WHISPER_BACKEND** (`faster`/`mlx`) / **WHISPER_MODEL**.

### Расширенный поиск (hybrid+)
- **LLM_METADATA** — извлекать продукт/тему/тип через LLM при индексации.
  *После переиндексации.*
- **SMART_FILTER** — умные фильтры из вопроса через LLM. *На лету.*
- **GRAPH_RAG** — граф-RAG для сводных вопросов. *На лету.*
- **GRAPH_MODE** — режим LightRAG (`mix`/`hybrid`/`local`/`global`/`naive`).

### Дообучение (fine-tuning)
- **USE_FINETUNED** — использовать дообученную модель для генерации. *На лету.*
- **FINETUNED_MODEL** — имя LoRA-адаптера в vLLM (по умолчанию `company-lora`).
- **FINETUNE_BASE** — базовая fp16-модель для дообучения. Пусто = берётся из
  `VLLM_MODEL` (без суффиксов квантизации). *На лету (применяется при следующем обучении).*

> Под каждым параметром этой группы в админке выводится **подробное описание и
> рекомендации**. У поля `FINETUNE_BASE` есть **выпадающий список fp16-моделей**
> (каталог под QLoRA) и кнопка **«🔄 Обновить список моделей»** — выбор подставляется
> в поле; пункт «из VLLM_MODEL (авто)» возвращает автоопределение. Эндпоинт списка:
> `GET /api/admin/finetune-models`.

### Доступ
- **ADMIN_TOKEN** — пароль админ-панели (секрет; в API не отдаётся).

После правок нажмите **«Сохранить настройки»**. «Сбросить к умолчаниям» возвращает
все значения и режим к стандартным.

## 4a. Добавление файлов через веб-интерфейс

В админке есть блок **«Добавить файлы (Excel и др.)»**: выберите один или несколько
файлов и нажмите «Загрузить» — они сохранятся в подпапку `uploads` папки
документов. Кнопка «Загрузить и переиндексировать» сразу запустит индексацию.
Принимается широкий набор форматов: PDF, DOCX/DOC, PPTX, XLSX/XLSM/XLS/CSV,
TXT/MD, HTML/MHTML, XML/JSON, SVG, ярлыки `.url`, письма Outlook `.msg`, чертежи
DXF/DWG и 3D-CAD STEP/IGES, изображения (JPG/PNG/…) и RAW-фото (CR2/…) с OCR, а
также архивы (ZIP/RAR/7Z/TAR — распаковываются, содержимое индексируется).
Excel/CSV разбираются построчно в формат «колонка: значение». Файлы
неподдерживаемых типов пропускаются с пометкой.

**Загрузка каталога целой папкой.** В том же блоке — выбор папки (кнопка выбора
каталога) и «📁 Загрузить папку». Структура подпапок сохраняется в папке
документов. Рассчитано на очень большой каталог (десятки тысяч файлов, проверено до
~50 000): загрузка идёт **порциями по 200 файлов** с индикатором прогресса и
кнопкой «Стоп»; неподдерживаемые типы отсеиваются ещё в браузере, чтобы не гонять
лишнее. Эндпоинт: `POST /api/admin/upload-folder` (files[] + paths[]). Опасные пути
(`..`) обезвреживаются, запись только внутрь папки документов.

## 4b. Сайты в базе знаний

Блок «Сайты в базе знаний» в админке: впишите до 50 адресов (по одному в строке)
и нажмите «Спарсить и добавить в базу». Страницы скачиваются, из них извлекается
текст (скрипты/меню/стили отбрасываются), сохраняется в подпапку `web/` папки
документов и индексируется. Список адресов запоминается.

Прямо в этом блоке:
- **Список «Сайты в базе»** — все добавленные сайты; у каждого кнопка **«Удалить»**,
  которая убирает сайт из списка, удаляет его файл из папки `web/` и его чанки из
  базы знаний (Qdrant) — без полной переиндексации.
- **Лог парсинга** — показывает ход скачивания (OK/ERR по каждому адресу) и
  последующей индексации; во время работы обновляется автоматически.

## 4c. Удалённые хосты (Qdrant / LightRAG по SSH)

Блок «Удалённые хосты» позволяет разворачивать и использовать компоненты на других
серверах. Профиль хоста: имя, ОС (Linux/macOS/Windows), адрес, SSH-порт, логин,
пароль, опционально `Qdrant URL` / `vLLM URL` / папка документов. Кнопки у каждого
хоста:

- **Развернуть Qdrant** — по SSH ставит/запускает контейнер Qdrant на хосте.
- **Развернуть LightRAG** — создаёт venv и ставит зависимости LightRAG на хосте.
- **Данные → / ←** — перенос папки документов на хост и обратно (SFTP), затем
  можно переиндексировать на месте.
- **Snapshot → / ←** — перенос готового индекса Qdrant между локальным и удалённым
  (через snapshot-API Qdrant).
- **Переключиться** — переводит приложение на удалённые `Qdrant URL`/`vLLM URL`
  (с сохранением локальной конфигурации); кнопка **«Вернуться на локальный»** —
  обратно. После переключения перезапустите сервис.

Ход операций виден в логе под списком. Ограничения и безопасность: SSH по
логину/паролю (нужна библиотека `paramiko`); на Linux требуется root или
passwordless sudo для установки Docker; на macOS/Windows Docker должен быть
предустановлен, а SSH-доступ включён; пароли хранятся в `remote_hosts.json` —
держите хосты в доверенной сети. Windows-деплой — best-effort (зависит от
настроек OpenSSH/PowerShell и Docker Desktop).

## 5. Операции обслуживания (админка)

Блок «Состояние и операции» показывает доступность Qdrant и LLM, число чанков,
статус задач и **живой лог** текущей операции (индексация, граф, дообучение,
установка) — он обновляется по ходу выполнения, а в конце показывает сводку
(`SUMMARY ... errors=N`). Кнопки:

- **Переиндексировать** — фоновый запуск `ingest.py`; инкрементально обновляет
  только изменённые файлы. Нужно после смены `DOCS_DIR`, чанкинга, включения
  LLM-метаданных или добавления документов.
- **Переиндексировать с нуля** — пересоздаёт коллекцию (`--reset`).
- **Применить модель LLM (рестарт vLLM)** — записывает `VLLM_*` и перезапускает
  контейнер vLLM с новой моделью (GPU).
- **Перезапустить сервис** — корректно завершает API; systemd (`Restart=always`)
  поднимает его заново. Нужно для параметров со scope «после рестарта».
- **Установить Qdrant + зависимости** — поднимает контейнер Qdrant (нужен Docker).
- **Установить LightRAG + зависимости** — pip-установка LightRAG в окружение (без
  построения графа).
- **Построить граф знаний** — установка LightRAG + `graph_rag ingest` (фоном);
  заменяет ручной запуск `gpu_variant/setup_hybrid.sh`.
- **Запустить дообучение** — фоновый пайплайн QLoRA (датасет + обучение адаптера).
- **Применить дообученную модель** — перезапуск vLLM с LoRA-адаптером.

### Калибровка по оценкам ответов

Блок «Калибровка по оценкам ответов» в админке использует оценки 👍/👎 из чата:
- **Проанализировать оценки** — показывает статистику (хорошие/плохие, средняя
  релевантность, плохие «без ответа» и «с ответом») и рекомендации по настройкам
  (порог `MIN_SCORE`, объём выборки `TOP_K_*`, `TEMPERATURE`, умные фильтры).
- **Применить рекомендации** — применяет предложенные изменения (с сохранением
  предыдущих значений для отката).
- **Откатить изменения** — возвращает настройки к значениям до последней калибровки.
- **Галка «Автоматически калибровать»** (`AUTO_CALIBRATE`) — каждые 10 новых оценок
  система сама применяет рекомендации.

Нужно ≥ 5 оценок для анализа. Логика эвристическая: частые «не знаю» при плохих
оценках → ослабить порог и расширить выборку; частые плохие при наличии ответа →
снизить температуру, включить умные фильтры, поднять порог.

### Обновления

Блок «Обновления» в админке:
- **Проверить обновления** — сравнивает локальную версию с GitHub (`git fetch`) и
  показывает, на сколько коммитов вы отстаёте и список изменений.
- **Обновить** — подтягивает последнюю версию (`git reset --hard origin/<ветка>`),
  переустанавливает зависимости и перезапускает сервис (через supervisor —
  systemd/launchd, без sudo). Работает, если установка делалась из GitHub.

### Резервное копирование и восстановление

Блок «Резервное копирование» в админке. Три области копирования:

- **Копия настроек** (`settings`) — только конфигурация: `runtime_config.json`, `.env`.
- **Копия служебных данных без каталога** (`service`) — настройки + операционные
  данные: журнал/оценки (`rag_logs.db`), тайминги (`ingest_stats.json`), SSH-хосты
  (`remote_hosts.json`), граф знаний (`graph_storage/`), LoRA-адаптер и датасет
  (`finetune/adapter`, `finetune/data`). Папка документов НЕ включается.
- **Полная копия** (`full`) — всё перечисленное + каталог документов (`DOCS_DIR`).

Каждый архив — `tar.gz` с `manifest.json`, где для каждого файла записан SHA-256;
рядом кладётся `<архив>.sha256`. **Контроль целостности** выполняется автоматически
сразу после создания (пересчёт хешей) и виден в списке. Архивы лежат в `backups/`,
их можно скачать (⬇️) или удалить (🗑). Векторный индекс Qdrant в копию не входит
(он производный) — после восстановления данных выполните переиндексацию.

**Восстановление**: загрузите архив, нажмите **«Проверить архив»** — выполняется
предварительный контроль целостности и состава (область, число файлов, проблемы);
кнопка «Восстановить» становится активной только если архив корректен. При
восстановлении сначала создаётся **страховочная копия** текущих служебных данных,
затем файлы распаковываются строго внутрь каталога проекта/`DOCS_DIR` (защита от
обхода путей). Эндпоинты: `POST /api/admin/backup/create|verify|restore|delete`,
`GET /api/admin/backup/list|download`. Создание и восстановление идут в фоне (статус
обновляется в блоке).

### База данных и кэш (SQLite / MySQL / PostgreSQL / Redis)

Блок «🗄 База данных и кэш» в админке. По умолчанию данные (журнал запросов, история
Телеграм, аналитика) хранятся в локальном **SQLite** (`rag_logs.db`) — без внешних
сервисов. При желании можно перенести их в **MySQL/MariaDB** или **PostgreSQL** —
например, чтобы несколько инстансов смотрели в общую БД.

Порядок переезда:

1. Создайте пустую базу на сервере СУБД (`CREATE DATABASE rag CHARACTER SET utf8mb4;`
   для MySQL или `CREATE DATABASE rag;` для PostgreSQL). Таблицы приложение создаст само.
2. В группе настроек **«База данных и кэш»** заполните хост/порт/логин/пароль/имя БД,
   нажмите **«Сохранить настройки»**.
3. **«Проверить MySQL/PostgreSQL»** — убедитесь, что подключение успешно.
4. Кнопки **«Копировать данные и настройки»** — копируют данные текущей активной БД в
   выбранную (таблицы приёмника очищаются — полная замена); активный бэкенд не
   меняется. Настройки (`runtime_config.json`) при этом сохраняются в таблицу
   `kv_store` приёмника (для переносимости между хостами).
5. Кнопки **«Мигрировать»** — копируют данные **и** переключают приложение на выбранную
   СУБД. После миграции выполните **«Перезапустить сервис»**. «Мигрировать обратно на
   SQLite» возвращает хранение в локальный файл.

Карточки статуса показывают для каждого бэкенда: настроен ли он, есть ли драйвер
(`PyMySQL` / `psycopg2`), доступен ли сервер, и какой бэкенд сейчас активен.
Записи в журнал устойчивы к сбою внешней БД — при недоступности запись пропускается,
а чат продолжает работать.

**Кэш Redis** (опционально, по умолчанию выключен): флажок `REDIS_ENABLED` + параметры
подключения. Кэширует тяжёлые агрегаты (статистика/аналитика) на 60 с; кэш
сбрасывается автоматически при каждом новом запросе, поэтому данные не устаревают.
Кнопка **«Очистить кэш Redis»** сбрасывает кэш вручную. Без Redis всё работает напрямую.

Драйверы (`PyMySQL`, `psycopg2-binary`, `redis`) входят в `requirements*.txt`; при
ручной установке поставьте их через «Переустановить окружение» или `pip install`.
Эндпоинты: `GET /api/admin/db/status`, `POST /api/admin/db/test|copy|migrate`,
`POST /api/admin/cache/clear`. Копирование/миграция идут в фоне (статус — в блоке).

### Переустановка и сброс

Блок «Переустановка и сборос» в админке:
- **Сброс данных** — кнопки очистки индекса Qdrant, графа, LoRA-адаптера и датасета,
  журнала, настроек, либо «Сбросить всё». `.env` и сам сервис сохраняются.
- **Переустановить окружение** — пересоздаёт `.venv` и переустанавливает
  зависимости, затем перезапускает сервис (фоновый `reinstall.sh`).
- **Полная переустановка** (DESTRUCTIVE) — кнопки «Полная переустановка
  (GPU-сервер)» и «Полная переустановка (Mac)»: останавливают сервис и контейнеры,
  удаляют окружение и ВСЕ данные (индекс, граф, адаптер, журнал) и разворачивают
  заново; `.env` сохраняется, сервис недоступен во время процесса. Те же действия
  доступны из shell: `sudo bash reinstall_server.sh` / `bash mac_variant/reinstall_mac.sh`
  (в скриптах можно пропустить вопрос подтверждения переменной `CONFIRM=yes`).
  Для GPU полной переустановке нужны права root (apt/systemd/docker).

## 6. Регулярное обновление индекса

Чтобы новые/изменённые файлы попадали в базу автоматически:

- **Apple**: пример задачи launchd — `reindex.plist` (каждый час). Установка
  описана в комментариях файла.
- **GPU/Linux**: cron, например ежечасно:
  ```
  0 * * * * cd /opt/rag && ./.venv/bin/python ingest.py >> /var/log/rag_reindex.log 2>&1
  ```

Индексация инкрементальна (хеш + mtime), поэтому повторные запуски дёшевы.

## 7. HTTP API

Базовый URL: `http://<сервер>:8000`. Админские эндпоинты требуют заголовок
`X-Admin-Token`, если задан `ADMIN_TOKEN`.

### Чат
`POST /chat` — тело `{"question": str, "history": [{role,content}], "filters": {...}|null}`.
Ответ — поток NDJSON: строки `{"type":"answer","text":...}` и финальная
`{"type":"sources","items":[...]}`. Поле `filters` (необязательно) задаёт явные
условия Qdrant, например `{"doc_category":"price"}`.

### Чат с приложенным документом
`POST /chat-doc` (multipart): `file`, `question`, `history` (JSON-строка). Парсит
файл, отбирает релевантные фрагменты и отвечает по ним; ответ — поток NDJSON, как
у `/chat`. Без индексации в Qdrant.

### Оценки и калибровка
- `POST /api/rate` — оценка ответа `{"id":N,"rating":1|-1|0}` (без токена, из чата).
- `GET /api/admin/recommendations` — анализ оценок и рекомендации.
- `POST /api/admin/apply-recommendations` / `POST /api/admin/rollback-calibration`.

### Мониторинг и данные
- `GET /health` — статус, модель, бэкенд.
- `GET /api/stats` — счётчики дашборда + число чанков.
- `GET /api/logs?limit=N` — последние запросы (по умолч. 100, макс. 1000).
- `GET /api/analytics` — агрегаты для графиков.

### Настройки и режим
- `GET /api/config` — схема полей + текущие значения (секреты маскируются).
- `POST /api/config` — изменить параметры (тело — словарь ключ/значение).
- `POST /api/config/reset` — сброс к умолчаниям.
- `GET /api/mode` — текущий режим и каталог режимов.
- `POST /api/mode` — `{"mode":"basic|meta|hybrid"}`.

### Администрирование
- `GET /api/admin/status` — доступность сервисов, число чанков, статус индексации.
- `POST /api/admin/reindex` — `{"reset": bool}`.
- `POST /api/admin/apply-llm` — перезапуск vLLM с текущей моделью.
- `GET  /api/admin/web-urls` / `POST /api/admin/ingest-web` — список сайтов и их
  парсинг с добавлением в базу (`{"urls":[...]}`, до 50).
- `POST /api/admin/install-qdrant` — поднять контейнер Qdrant (фоном).
- `POST /api/admin/install-lightrag` — установить LightRAG в окружение (фоном).
- `POST /api/admin/build-graph` — установка LightRAG и построение графа (фоном).
- `POST /api/admin/finetune` — запуск пайплайна дообучения (фоном).
- `POST /api/admin/apply-finetuned` — перезапуск vLLM с LoRA-адаптером.
- `POST /api/admin/reset` — `{"targets":["index","graph","adapter","logs","settings"|"all"]}`.
- `GET  /api/admin/check-updates` — сравнение версии с GitHub.
- `POST /api/admin/update` — обновление из GitHub + перезапуск (фоном).
- `POST /api/admin/reinstall-env` — переустановка окружения (фоном).
- `POST /api/admin/reinstall-full` — `{"kind":"server"|"mac"}` полная переустановка (фоном).
- `GET  /api/admin/db/status` — статус бэкендов БД (sqlite/mysql/postgresql) и кэша Redis.
- `POST /api/admin/db/test` — `{"backend":"mysql|postgresql|sqlite"}` проверка подключения.
- `POST /api/admin/db/copy` — `{"target":...}` копировать данные и настройки в СУБД (фоном).
- `POST /api/admin/db/migrate` — `{"target":...}` копировать и переключить активную БД (фоном).
- `POST /api/admin/cache/clear` — очистить кэш Redis.
- `POST /api/admin/restart` — перезапуск сервиса.
- `GET /api/admin/remote/hosts`, `POST /api/admin/remote/{save,delete,deploy,transfer,switch,restore}`
  — удалённые хосты: профили, деплой (qdrant/lightrag), перенос (docs/snapshot,
  push/pull), переключение бэкенда и возврат.

Пример запроса с токеном:

```bash
curl -X POST http://localhost:8000/api/mode \
  -H 'Content-Type: application/json' -H 'X-Admin-Token: ВАШ_ТОКЕН' \
  -d '{"mode":"hybrid"}'
```

## 8. Командная строка

Из корня проекта при активном `.venv`:

- `python ingest.py` — индексация (инкрементальная).
- `python ingest.py --reset` — переиндексация с нуля.
- `python -m graph_rag ingest` — построение графа знаний (hybrid+).
- `python compare.py [questions.txt|"вопрос1" "вопрос2"]` — сравнение ответов
  вектор-пайплайна и LightRAG на одинаковых вопросах.
- `python finetune/build_dataset.py` / `python finetune/train_lora.py` — сбор
  датасета и QLoRA-дообучение (или единым скриптом `bash finetune/run_pipeline.sh`).

## 9. Безопасность и эксплуатация

- **Обязательно задайте `ADMIN_TOKEN`** — иначе админка и операции доступны всем в
  сети. Менять токен можно в админке (поле «Токен администратора» в группе «Доступ»).
- Для внешнего доступа закройте сервис reverse-proxy (Caddy/Nginx) с TLS и
  авторизацией или ограничьте firewall’ом локальной сетью.
- Данные не покидают сервер: модели, индекс и документы — локальные.
- Журнал запросов хранится в `rag_logs.db` (SQLite) рядом с приложением.

# Документация по использованию

Руководство по повседневной работе с системой: веб-панель для сотрудников и
администратора, режимы работы, справочник всех настроек, операции обслуживания,
HTTP API и утилиты командной строки.

---

## 1. Запуск и доступ

После установки сервис доступен по адресу `http://<ip-сервера>:8000`.

- На **GPU-сервере** API работает как systemd-сервис `rag-api` (автозапуск).
  Управление: `bash gpu_variant/manage.sh {status|logs|restart|stop|start}`.
- На **Apple** запуск вручную: `uvicorn app:app --host 0.0.0.0 --port 8000`
  (можно оформить в `launchd`).

Сотрудникам достаточно открыть адрес и пользоваться вкладкой **«Чат»**. Остальные
вкладки — для администратора/аналитики.

---

## 2. Веб-панель

Боковое меню содержит разделы (дашборд, схема и тесты, чат, журнал, аналитика,
аналитика+, система, бенчмарк, администратор). Внизу — переключатель темы
оформления (Тёмная / Cyberpunk 2077 / Лето), текущий режим работы и активная
модель.

### 2.1. Дашборд
Сводные счётчики: всего запросов, за сегодня, доля отвеченных, средняя
релевантность, средняя задержка, **среднее время поиска и генерации** (этапы
запроса), число чанков в базе; плюс последние запросы. Кнопка «Обновить»
перечитывает статистику.

Блоки **«Последние запросы (веб-чат)»** и **«Последние запросы в Телеграм»** —
отдельные таблицы по каналам, с оценкой (👍/👎) и комментарием к ответу.

Блок **«Текущие запросы»** показывает в реальном времени (автообновление ~1.5 с)
всё, что система обрабатывает прямо сейчас: вопросы из веб-чата и Телеграм с
текущей стадией (поиск → генерация ответа), разбор приложенных файлов, парсинг
справочника компании и идущие фоновые задачи (индексация, построение графа,
бенчмарк, проверка данных, бэкап и т. п.) — с подписью стадии и временем
выполнения. Источник — `GET /api/activity` (реестр `activity.py` + активные джобы
`admin_ops`). Завершённые задачи ещё несколько секунд видны с отметкой ✓/✖.

### 2.2. Схема работы и тесты
**Живая схема** всей связки, построенная по реальным настройкам и статусам (данные из
`GET /api/config` и `GET /api/system`, автообновление каждые ~7 с). Узлы: источники
(документы/сайты/загрузка) → ингест → эмбеддер → Qdrant; вопрос → режим/движок →
поиск+реранк **/** LightRAG **/** KAG → LLM → ответ; инфраструктура — БД, Redis,
удалённые хосты. Для каждого узла показаны **статус** (цветная точка: работает / готов /
нужен-не готов / недоступен / выключен) и **ключевые настройки** (модель и устройство
эмбеддера, размер чанка и batch, top-k/порог, режим графа, шаги KAG, бэкенд и модель LLM,
число чанков в Qdrant, сущности/связи графа и т. п.).

**Доступность связей зависит от текущего режима/движка:** активные связи подсвечены и
анимированы (зелёный «поток»), доступные, но не используемые — серый пунктир, недоступные
(офлайн-компонент) — приглушены. Например, при движке «Векторный» горит ветка поиска, при
«LightRAG» — ветка графа, при «KAG» — мультихоп с опциональной подкачкой знаний из графа;
связь «каталог → БД» активна при каталоге в PostgreSQL; связи Redis — при включённом кэше.
Переключатель **«анимация»** отключает движение, кнопка **«Демо-прогон ответа»** по шагам
подсвечивает путь ответа для текущего движка.

Ниже — **«Тестирование компонентов»**: кнопка «Запустить тесты» проверяет каждый
компонент (настройки, Qdrant, эмбеддер, реранкер, LLM-генерация, папка документов,
журнал, граф, дообучение), показывает результат по каждому (OK/FAIL + детали),
общий вердикт и лог. Граф и дообучение считаются опциональными и не влияют на
общий результат. Эндпоинт: `POST /api/admin/selftest` (под токеном).

### 2.3. Чат
Основной интерфейс. Сотрудник задаёт вопрос — ответ приходит потоково, со
ссылками на источники (файл и, где есть, страница/слайд). Если в документах нет
ответа, система честно сообщает об этом, а не выдумывает.

**Анимация конвейера.** Над каждым ответом в реальном времени показывается
сворачиваемый блок «⚙️ Конвейер ответа» — какие модули отрабатывают и с какими
параметрами: эмбеддинг запроса (модель, устройство), фильтр (тип, значения),
векторный поиск Qdrant (top-k, число кандидатов), лексика BM25, реранк (модель,
порог, сколько оставлено), сборка контекста (фрагментов, символов) и генерация LLM
(бэкенд, модель, температура, растущий счётчик символов). У каждого этапа — время
выполнения; активный этап подсвечивается. Попадание в кэш Redis показывается
отдельным этапом. Для приложенного документа и движка LightRAG — свои этапы.

**Подробности ответа.** Галка «Подробно об источниках и алгоритме формирования
ответа» под полем ввода добавляет к каждому ответу сворачиваемый блок: движок и
режим, модель и бэкенд, тайминги (поиск/генерация/всего), параметры поиска и
список использованных фрагментов (источник, страница, score, отрывок текста).
Состояние галки запоминается в браузере.

**Оценка ответа.** Под каждым ответом — кнопки 👍 / 👎. Оценки копятся в журнале,
видны в «Аналитике» (доля «хорошо», удовлетворённость) и используются для
калибровки настроек (см. раздел «Калибровка по оценкам» в админке).

**Визуальные источники.** Кроме текстовых ссылок, в блоке «Источники» показываются
сами артефакты: миниатюры изображений и RAW-фото, превью чертежей DXF/DWG (рендер),
плеер аудио, кадр-превью видео с кнопкой ▶ (открывает видео с перемоткой на нужный
момент) и ссылкой «⤓ фрагмент» (вырезает отрезок видео по таймкодам сегмента).
Превью строятся «лениво» при первом обращении и кешируются в `previews/` —
переиндексация не нужна; тайминги видео берутся из транскрипции (после переиндексации
аудио/видео в их чанках появляются `t_start`/`t_end`). Эндпоинты:
`GET /api/media/file|thumb|clip|info?source=…`. Для чертежей нужен `matplotlib`
(рендер ezdxf), для кадров/фрагментов — `ffmpeg`.

**Голосовой ввод и ответ.** Кнопка 🎤 у поля ввода записывает речь и распознаёт её
**локально** через тот же Whisper, что и для аудио/видео (данные не покидают сервер;
эндпоинт `POST /api/transcribe`): нажмите 🎤, говорите, нажмите ещё раз — текст
подставится в поле. Галка «🔊 Озвучивать ответы голосом» включает озвучку ответов
(синтез речи браузера, голос ОС, русский); у каждого ответа есть кнопка 🔊 для
ручного воспроизведения/остановки. Важно: доступ к микрофону браузеры дают только в
защищённом контексте — по `https://` или с `localhost`; при открытии по `http://<ip>`
запись будет недоступна (озвучка ответов работает и по http). Для голосового ввода в
сети поднимите HTTPS (reverse-proxy) или открывайте с самого сервера.

**Сессии и история.** Каждая сессия — отдельный чат со своим `session_id` (виден
как «чат #…» в шапке); все запросы помечаются им в журнале. Кнопка **«🆕 Новый
чат»** очищает окно и начинает новый чат (прежний уже сохранён в журнале). Кнопка
**«💾 Сохранить историю»** выгружает переписку текущего чата в Markdown-файл
(вопросы, ответы и источники). История одного чата также доступна по API:
`GET /api/chat-history?session_id=…`. Очистить историю всех чатов и статистику
можно в разделе «Администратор» (кнопка «🗑 Очистить историю всех чатов и
статистику», эндпоинт `POST /api/admin/clear-history`).

**Приложить документ к вопросу.** Кнопка 📎 рядом с полем ввода позволяет
«подложить» файл (Excel и др.) прямо к конкретному вопросу: файл парсится на лету,
самые релевантные строки подаются в контекст LLM — без добавления в общую базу и
без переиндексации. Удобно для разовых вопросов по конкретной таблице. Если файл
нужен постоянно — загрузите его в админке и переиндексируйте.

### 2.4. Журнал
**Объединённый журнал** показывает обращения и из веб-чата, и из Телеграм — каждое
помечено каналом (🌐 Веб / ✈️ Телеграм), с **вопросом**, **текстом ответа**
(колонка «Ответ»; полный текст — во всплывающей подсказке), статусом, **оценкой**
(👍/👎) и **комментарием** пользователя. Источник — `GET /api/journal` (web+tg).
Текст ответа сохраняется в колонке `answer` таблиц `requests`/`tg_requests`. Оценить и
прокомментировать ответ можно прямо под ответом в чате (кнопки 👍/👎 и
«💬 Комментировать», эндпоинты `/api/rate` и `/api/comment`) и в Телеграм (инлайн-
кнопки под ответом бота).

### 2.5. Аналитика
Графики (Chart.js): запросы по дням, распределение по категориям, топ
цитируемых источников, распределение задержек, облако частых слов из вопросов.
Отдельный блок **«✈️ Частые слова в вопросах Телеграм»** — частотный анализ по
вопросам бота (отдельно от веб-чата). Помогает понять, что спрашивают чаще всего и
каких документов не хватает.

### 2.6. Система
Подробная информация и статистика по компонентам:
- **Qdrant** — число чанков, сегментов, размерность вектора и метрика, статус
  коллекции, поля payload, разбивка чанков по категориям, сколько чанков получили
  LLM-метаданные.
- **LightRAG (граф)** — установлен ли пакет, построен ли граф, число сущностей,
  связей, чанков и документов, размер хранилища, текущий движок и режим графа.
- **Дообучение (LoRA)** — установлены ли зависимости, готов ли адаптер и активен
  ли он, число пар в датасете, размер адаптера, базовая модель.
- **Hybrid+ и движки** — текущий режим и состояние флагов (метаданные, умные
  фильтры, граф, авто-фильтр), а также сколько ответов дано каждым движком
  (векторный / граф / LightRAG / KAG).
- **KAG (знание-усиленная генерация)** — сводка: активен ли KAG как движок, его
  параметры (декомпозиция, макс. шагов, фрагментов на шаг, итоговый контекст,
  взаимное индексирование), знания графа (использовать ли граф, построен ли он, режим
  графа, ссылки в ответе), параметры поиска, которые использует мультихоп KAG
  (`MIN_SCORE`/`TOP_K_RETRIEVE`/`TOP_K_RERANK`, температура), бэкенд/модель LLM, число
  ответов через KAG и схема конвейера. Предупреждает, если «использовать граф»
  включено, но граф не построен.
- **Время обработки (индексация)** — длительность последней индексации, число
  обработанных файлов и чанков, среднее время на файл, суммарное время по всем
  файлам и когда обновлялось (по данным `ingest_stats.json`).
- **Базы данных (SQLite / MySQL / PostgreSQL)** — по каждому бэкенду: статус
  (доступна / не настроена / нет драйвера / недоступна), какой активен, версия
  сервера, размер БД, число записей (запросы web, запросы Телеграм, пользователи
  бота, ключи настроек в `kv_store`), адрес/файл. Показывает данные и в неактивных,
  но доступных БД (например, после копирования).
- **Кэш Redis** — статус и расширенная статистика: версия, занятая память, число
  ключей кэша приложения и всего ключей в БД Redis, число клиентов, аптайм, процент
  попаданий кэша (hit rate), hits/misses, вытеснённые ключи. Если выключен или
  недоступен — об этом сообщается.
- **Каталог документов** — расширенная информация по файлам папки знаний: сводка
  (всего файлов, проиндексировано/нет, объём, разбивка по типам) и таблица с
  поиском по имени: файл, тип, размер, число чанков, статус (в базе/нет) и дата
  изменения. Есть **сортировка** (по имени, дате, размеру или числу чанков —
  кликом по заголовку столбца или выпадающим списком, с переключением направления)
  и **пагинация** (выбор размера страницы 50/100/200/500 и кнопки «Назад/Вперёд»).
  **Файлы с ошибками** подсвечиваются: после запуска «Проверить каталог с данными»
  в каталоге появляется счётчик «С ошибками», у проблемных строк — пометка
  «⚠ ошибка» с описанием причины (пустой/нечитаемый файл и т. п.), есть фильтр
  «только с ошибками» и сортировка «ошибки сверху» (клик по заголовку «Статус»).
  **Время обработки**: в сводке — карточка «Общее время обработки» и строка о
  последней индексации (длительность, число файлов, среднее на файл); в таблице —
  столбец «Время обр.» по каждому файлу с сортировкой (клик по заголовку). Данные
  берутся из `ingest_stats.json`, который пишет `ingest.py` при индексации.
  **Способ извлечения текста**: каждый файл помечается — 🎤 транскрибация
  (аудио/видео → Whisper), 🔍 OCR (изображения/RAW-фото) или 🛠 спец-инструмент
  (DWG/STEP/IGES, `.doc`, `.msg`, архивы). В сводке — счётчики «Транскрибировано» и
  «Распознано»; фильтр «Способ» позволяет отобрать транскрибированные/распознанные/
  и т. д. Кнопка 👁 в строке **раскрывает превью и содержимое файла**: миниатюра
  изображения/RAW-фото, рендер чертежа DXF/DWG, первая страница PDF, кадр видео с
  кнопкой ▶ (открывает плеер), проигрыватель аудио; ниже — извлечённый текст
  (транскрипция/распознанное) для проиндексированных файлов и ссылка «скачать
  оригинал». Превью доступно и для ещё не проиндексированных визуальных файлов
  (`GET /api/media/thumb|file|clip`, `GET /api/admin/file-text`).
  Эндпоинт каталога: `GET /api/admin/files-catalog?limit=&offset=&sort=name|date|size|chunks|proc|error&order=asc|desc&q=&only_errors=&method=transcribed|recognized|ocr|tool|text`.
- **Проверка каталога данных** — кнопка «Проверить каталог с данными» сканирует
  всю папку и выдаёт: сводку (ок/медиа/пустые/неподдерживаемые/таймаут/ошибки),
  список неподдерживаемых типов с рекомендациями (конвертация, OCR, распаковка,
  установка `xlrd` и т.п.) и таблицу проблемных файлов (файл → проблема → решение).
  Разбор файлов идёт **параллельно** в пуле процессов (до 8, по числу ядер) — это
  кратно быстрее на больших каталогах; узкое место — нативный парсинг (PDF/Office),
  поэтому ускорение даёт именно многопроцессность, а не смена языка. Лимит времени
  на файл берётся из `FILE_PARSE_TIMEOUT` (превышение → статус «таймаут», файл не
  блокирует проверку). Медиа/архивы/OCR при проверке не разбираются.
  Эндпоинт: `POST /api/admin/check-data`.

### 2.6a. Загрузка сервера
Отдельный раздел с текущей загрузкой хоста в реальном времени (галка
«автообновление» опрашивает каждые 2 с). Источник метрик — `psutil` (плюс
`nvidia-smi` для GPU). Показывает:
- **CPU** — общая загрузка и по каждому ядру, число потоков/ядер, частота,
  load average;
- **Память и swap** — занято/всего/свободно с индикатором заполнения;
- **Диски** — таблица по всем разделам (точка монтирования, ФС, занято/всего/
  свободно, процент);
- **GPU** — для NVIDIA: загрузка ядра и видеопамяти, использовано/всего МБ,
  температура, питание (Вт), вентилятор; для Apple Silicon — встроенный GPU с
  единой памятью (живую загрузку без root не получить); если GPU нет — отметка,
  что вычисления на CPU;
- **Сеть и аптайм**, а также **топ процессов по памяти** (PID, имя, RSS, CPU%).

Эндпоинт: `GET /api/server-load`. Если `psutil` не установлен — отдаётся
сокращённый набор (CPU-ядра, load average, память из `/proc`, корневой диск).

### 2.6b. Телеграм-бот
Доступ к базе знаний через Telegram. Бот работает на том же RAG-конвейере, что и
веб-чат (выбранный движок — Векторный/LightRAG/KAG, доп. поиск при отсутствии ответа),
и отвечает со ссылками на источники.

**Голос и файлы (раздел «Телеграм» → «🎙 Голосовые сообщения, ответы голосом и файлы»).**
Все настройки доступны прямо в этом разделе (с подробными описаниями и рекомендациями) и
сохраняются на лету; там же — индикатор: бэкенд Whisper, наличие `ffmpeg` и доступный
TTS-движок.
- **Распознавание голосовых** (`TELEGRAM_VOICE_IN`, по умолчанию вкл): голосовое сообщение
  скачивается и **распознаётся локальным Whisper** (модель — группа «Транскрибация»),
  распознанный текст обрабатывается как обычный вопрос (бот эхом показывает вопрос).
- **Распознавание приложенных файлов** (`TELEGRAM_FILES`, по умолчанию вкл): приложите файл
  (PDF, DOCX, XLSX, изображение и др.) с **вопросом в подписи** — бот извлечёт из него текст
  (с OCR для картинок) и ответит по содержимому именно этого файла (в базу не добавляется);
  без подписи присылает распознанный фрагмент.
- **Ответ голосом** (`TELEGRAM_VOICE_OUT`): на голосовой запрос бот, помимо текста, присылает
  **озвученный ответ** — синтез речи движком `TTS_ENGINE` (`auto`/`piper`/`say`/`espeak`/`off`),
  голос/модель — `TTS_VOICE`. Аудио кодируется в OGG/Opus через **ffmpeg**. Если TTS
  недоступен — отправляется только текст.
- **Выбор голоса.** Под движком — выпадающий список **доступных голосов** (кнопка «↻ Обновить
  список»): для `espeak` — много тембров одного языка (мужские `+m1…+m7`, женские `+f1…+f5`,
  хриплый, шёпот) и другие языки; для `say` (macOS) — системные голоса с локалями; для `piper` —
  найденные модели `.onnx` (из `DOCS_DIR/piper`, `~/piper`, `/opt/piper` и др.). Можно выбрать из
  списка или вписать значение вручную. Источник списка — `GET /api/admin/tts/voices`.

Рекомендации по TTS: `piper` — лучшее качество и естественный русский (нужна модель `.onnx`),
`say` — встроен в macOS, `espeak` — есть на любом Linux, но «робот-голос». Голосовые и файлы
требуют установленного `ffmpeg` и бэкенда Whisper.

**Обучение бота пользователями (раздел «Телеграм» → «🎓 Обучение бота»).** Администратор
может **разрешить** выбранным пользователям пополнять базу знаний через Telegram (галка
«разрешено» напротив пользователя). Такой пользователь командой **`/train`** входит в
**режим обучения** и присылает документы (PDF, DOCX, XLSX, фото и др.) — каждый файл
скачивается в **отдельную папку на пользователя** (`DOCS_DIR/telegram/<chat_id>/`),
распознаётся (с OCR) и **добавляется в базу знаний** (Qdrant); бот присылает статус
(«📥 Скачиваю…», «🔎 Распознаю…», «✅ добавлено N фрагментов»). Командой **`/ask`**
пользователь выходит из режима и снова задаёт вопросы. В веб-панели по каждому
пользователю видно число файлов, можно **посмотреть список файлов**, удалить **отдельный
документ** или **все документы пользователя**, а также **все документы, добавленные через
Телеграм**. **При выдаче разрешения** бот автоматически отправляет пользователю подробную
инструкцию (как работает система и как обучать: `/train` → присылать документы → `/ask`);
рядом с каждым пользователем есть кнопка **«📨 Инструкция»** — отправить её повторно.
Эндпоинты: `GET /api/admin/telegram/train-users`, `POST .../train-allow` (при `allow=true`
шлёт инструкцию), `POST .../train-instruction`, `GET .../train-files`,
`POST .../train-delete`, `POST .../train-delete-all`.

**Подключение.** Создайте бота у **@BotFather** в Telegram, получите токен и
вставьте его в «Администратор» → группа **«Телеграм-бот»** (`TELEGRAM_BOT_TOKEN`,
секретное поле). Нажмите «Сохранить настройки», затем в разделе **«Телеграм»** —
**«Перезапустить бота»**. Бот стартует автоматически при запуске сервиса, если
токен задан.

**Сообщения пользователям.** Блок **«✉️ Отправить сообщение пользователям»** позволяет
из веб-интерфейса разослать текст от имени бота: **выбранным** (отметьте галочками в
таблице пользователей; есть «выбрать всех»), **всем подтверждённым** или **всем**
(кроме заблокированных). Показывается результат «доставлено N / всего» и число
недоставленных (например, кто остановил бота). Эндпоинт: `POST /api/admin/telegram/send`
(`{text, chat_ids?|scope: 'all'|'approved'}`). Рассылка идёт с паузами под лимиты
Telegram и видна в «Текущих запросах».

**Сопоставление с сотрудниками.** В таблице пользователей есть колонка **«Сотрудник»**:
кнопкой **«🔗 Сопоставить»** открывается поиск по справочнику компании (раздел
«Структура компании»), и выбранный сотрудник привязывается к Телеграм-пользователю
(хранится e-mail как ключ + ФИО и «должность · отдел · локация» для отображения).
Привязку можно изменить (✏️) или снять (✕). Поиск идёт через `GET /api/admin/org/list`,
привязка — `POST /api/admin/telegram/map-employee` (пустые поля снимают связь; данные
в таблице `tg_users`: `emp_email`/`emp_name`/`emp_info`). Требуется заранее
синхронизированный справочник.

**Подтверждение пользователей.** По умолчанию новый пользователь, написавший боту,
попадает в **«Запросы на доступ»** и получает сообщение об ожидании. Администратор
в разделе «Телеграм» нажимает **«Подтвердить»** (бот уведомит пользователя) или
**«Отклонить»** (блокировка). Подтверждённые могут задавать вопросы; список всех
пользователей (подтверждён/заблокирован/ожидает) — там же, с действиями
заблокировать/разблокировать. Галка **«авто-подтверждение»** (`TELEGRAM_AUTO_APPROVE`)
открывает доступ всем без ручного подтверждения.

**Прокси.** Если Telegram заблокирован в вашей сети, в группе «Телеграм-бот» задайте
`TELEGRAM_PROXY` — бот будет ходить к `api.telegram.org` через прокси. Поддерживаются
`socks5://`, `socks5h://`, `http://`, `https://` (можно с `user:pass@`). **Важно:**
бот работает по HTTP Bot API, поэтому **MTProto-прокси (tg://proxy-ссылки) не
подходят** — они для клиентов Telegram; укажите обычный SOCKS5/HTTP-прокси. Текущий
прокси виден в карточке статуса бота. После изменения — «Перезапустить бота».

**Workflow бота (анимированный граф).** Блок **«🔀 Workflow бота»** показывает путь
обработки сообщения в виде анимированного графа: приём → проверка доступа → тип
(текст / голос → STT / файл → OCR / команда обучения) → RAG-ответ (с синонимами,
прайс-папкой и внешними API) → отправка → озвучка → оценка → журнал. По графу
бежит подсветка активного этапа; ветки, отключённые в настройках (распознавание
голоса/файлов, ответ голосом, кнопки оценки), приглушены. Кнопка ⏸/▶ — пауза/запуск.

**Структура формирования ответа.** Как и в веб-чате, бот добавляет к ответу
**«структуру формирования ответа»** — этапы конвейера (эмбеддинг → фильтр →
векторный поиск → BM25 → реранк → сборка контекста → генерация; либо шаги KAG,
либо движок-граф) с ключевыми параметрами и временем каждого этапа. Управляется
галкой **«Показывать структуру формирования ответа»** (`TELEGRAM_PIPELINE`, по
умолчанию включено); выключите, чтобы бот присылал только ответ и источники.

**Вывод ответа LLM.** Галка **«Выводить ответ LLM»** (`TELEGRAM_SHOW_ANSWER`, по
умолчанию включено) управляет отправкой самого текста ответа модели. Выключите,
если нужно присылать только источники и/или структуру формирования ответа (например,
для отладки поиска); при выключенном ответе голосовой ответ не отправляется, а если
выключены и ответ, и источники, и структура — бот пришлёт короткую отметку об
обработке. Обе галки — в разделе **«Телеграм»** (блок «🎙 Голосовые сообщения…»)
и в группе настроек «Телеграм-бот».

**Превью источников.** Галка **«Присылать превью источников»** (`TELEGRAM_PREVIEWS`,
по умолчанию включено) добавляет к ответу бота визуальные превью источников — так же,
как карточки-превью в веб-чате: миниатюры картинок, RAW-фото и чертежей (DXF/DWG),
первую страницу PDF, кадр видео (с таймкодом) и сам аудио-файл. Источники
дедуплицируются, число превью на ответ ограничено `TELEGRAM_PREVIEW_MAX` (по умолчанию
4). Превью строятся «лениво» и кешируются в `previews/`; нужны Pillow/ffmpeg и доступ
к папке документов. Отправляются через `sendPhoto`/`sendAudio`.

**Отдельная история.** Запросы и ответы из Телеграм хранятся **отдельно** от
веб-чата (таблицы `tg_users`/`tg_requests`) и показываются в блоке «История и
запросы из Телеграм» со своей статистикой (всего/сегодня/пользователей/доля
ответов/задержка) и кнопкой очистки. Эндпоинты: `GET/POST /api/admin/telegram/*`
(status, restart, users, approve, block, unblock, requests, clear-history).

### 2.6a. Загрузка сервера — история и рекомендации по железу
В разделе **«Загрузка сервера»**, помимо метрик в реальном времени, есть блок
**«Статистика загрузки»** за **час / день / неделю / месяц / год** (среднее и пик
по CPU, памяти, swap, GPU, видеопамяти, диску) и блок **«Рекомендации по железу»**.
Фоновый сборщик (`monitor.py`) снимает показатели раз в минуту и складывает их в
таблицу `server_samples` (выборки старше ~13 месяцев удаляются автоматически).
Рекомендации формируются автоматически по накопленной статистике: где есть запас,
а где пора усиливать конфигурацию (больше ядер CPU, ОЗУ, VRAM, диск). Эндпоинты:
`GET /api/server-load` (текущая загрузка), `GET /api/server-history` (история +
рекомендации). История доступна при установленном `psutil`.

### 2.6b. Структура компании
Раздел **«Структура компании»** — справочник сотрудников, который **автоматически
парсится по URL раз в час**. Источник должен отдавать выгрузку в формате
**SpreadsheetML** (Excel 2003 XML, как у 1С/СБИС) с колонками: работает, работает до,
фамилия, имя, отчество, должность, отдел, тел. рабочий/внутренний/мобильный,
электропочта, поставщики. Поле «отдел» очищается от HTML-мусора (теги `<span>`,
ссылки на базы знаний, `&nbsp;`), а код локации (`(МСК)`, `(СПБ)` …) выносится в
отдельную колонку. Укажите URL, включите **«ежечасную синхронизацию»** и/или нажмите
**«Синхронизировать сейчас»**; есть поиск (ФИО, должность, телефон, e-mail,
поставщик), фильтр по отделу и галка «только работающие». Данные полностью
заменяются при каждой синхронизации (таблица `org_employees`). Расписание крутит
`monitor.py`. Эндпоинты (под `X-Admin-Token`): `GET/POST /api/admin/org/config`,
`POST /api/admin/org/sync`, `GET /api/admin/org/list`, `POST /api/admin/org/clear`.

**Индексация в базу знаний.** После **каждой** синхронизации карточки сотрудников
(ФИО, должность, отдел, локация, все телефоны, e-mail, поставщики, статус)
добавляются в векторный индекс (`org_index.py`), чтобы ассистент отвечал на вопросы
вроде «телефон Иванова», «кто руководит отделом ТКП», «почта менеджера в Казани».
Индексация **инкрементальная**: новые и изменённые карточки переиндексируются,
удалённые из справочника — убираются из базы. **Каждый сотрудник — отдельная
сущность** (персональная точка Qdrant и персональный источник в ссылках, например
«Иванов Иван — Инженер · Отдел ТКП»), поэтому в графе «База знаний (Obsidian)»
сотрудники отображаются **отдельными узлами, сгруппированными по отделам** (отдел
выступает кластером) — получается интерактивная орг-структура. Очистка справочника
удаляет эти карточки и из индекса.

### 2.6c. Синонимы
Раздел **«Синонимы»** — словарь, повышающий качество поиска и ответов. Для каждого
слова задаётся несколько синонимов (через запятую); слова в группе считаются
**взаимозаменяемыми**. Когда учёт синонимов **включён** (галка вверху раздела):

* **Понимание вопросов** — если в вопросе встретилось слово из группы, остальные
  слова группы добавляются к поисковому запросу (полнее находятся документы — и по
  лексическому BM25, и по плотному эмбеддингу).
* **Формирование ответов** — модели передаётся краткая подсказка о равнозначности
  встретившихся слов, чтобы ответ учитывал синонимию.

Сопоставление идёт по границам слова (с поддержкой кириллицы), поддерживаются
многословные синонимы. Изменения применяются сразу (кэш поиска учитывает версию
словаря). Эндпоинты (под `X-Admin-Token`): `GET /api/admin/synonyms`,
`POST /api/admin/synonyms/{config,add,update,delete,clear}`. Хранилище — таблица
`synonyms`, флаг включения — в `kv_store` (`syn.enabled`).

> **Внутренний URL и DNS.** Если адрес выгрузки внутренний (например
> `http://in.vodokomfort.ru/export/excel`), сервер RAG должен находиться во
> внутренней сети и уметь разрешать это имя. Ошибка «Не удалось разрешить имя
> хоста (DNS)» означает, что хост не резолвится из среды RAG. В Docker пропишите
> внутренний DNS в compose (`dns: [IP_внутреннего_DNS]`) или сопоставьте имя с
> адресом через `extra_hosts: ["in.vodokomfort.ru:IP"]`, либо укажите в URL
> IP-адрес сервера выгрузки.

### 2.6d. База знаний — граф (Obsidian)
Раздел **«🔮 База знаний (Obsidian)»** показывает текущую проиндексированную базу в
виде интерактивного графа, как граф-вью Obsidian: узлы — **файлы** и **категории**,
связи — принадлежность файла категории; категории образуют кластеры. Размер узла-
файла пропорционален числу его фрагментов, узла-категории — числу файлов; цвет —
по категории. Граф рисуется на canvas (force-раскладка, без внешних зависимостей):
колесо — масштаб, перетаскивание узла/фона, **клик по узлу** открывает справа панель
с деталями (категория, число фрагментов/страниц, путь) и **фрагментом текста файла**
(подгружается из индекса). Есть поиск с подсветкой и переключатель подписей.
Граф растягивается **на всю ширину экрана** (панель деталей — выезжающая справа,
закрывается крестиком). Сборка идёт **в фоне с прогресс-баром** (показывает стадию и
число обработанных фрагментов из общего числа точек Qdrant). Результат **кэшируется**
(TTL ~5 минут): повторное открытие отдаёт граф мгновенно с пометкой «из кэша»; кнопка
**«🔄 Пересобрать»** строит граф заново. Данные собираются одним проходом по Qdrant
(`admin_ops.kb_graph` — фоновая задача; эндпоинты `GET /api/admin/kb-graph` и
`GET /api/admin/kb-graph/status`); число файловых узлов ограничено (по умолчанию 400
самых крупных) для производительности.

**ИИ-оценка базы знаний (Автокалибровка).** Кнопка **«Оценить базу знаний»** запускает
аудит: `kb_eval.py` профилирует весь индекс (объём, категории, типы файлов, покрытие
метаданными, длины чанков, дубликаты, примеры фрагментов, текущие настройки) и
передаёт профиль модели, которая возвращает оценку и конкретные рекомендации по
контенту, индексации (chunk_size/overlap, метаданные) и настройкам поиска (MIN_SCORE,
TOP_K, температура, движок/режим, синонимы). Идёт фоном с прогрессом, результат
кэшируется. Эндпоинты: `POST /api/admin/calib/eval-kb`, `GET .../eval-kb/status`.

### 2.7. Аналитика+
Расширенная аналитика по компонентам с графиками (Chart.js). В каждом разделе, кроме
графиков, есть **прогресс-бары** с наглядными долями: Qdrant — покрытие чанков
LLM-метаданными (product/topic/doc_type) и распределение по категориям; Граф —
величины сущностей/связей/чанков/документов относительно максимума и охват базы
графом; Дообучение — готовность по этапам (датасет · параметры · адаптер, зелёный
при 3/3) и наполнение датасета; Тайминги — доля поиска и генерации в общем времени
ответа и бенчмарк компонентов относительно самого медленного. Ниже — графики:
- **Qdrant** — чанков по категориям (кольцевая), по типам файлов (столбцы),
  покрытие LLM-метаданными (product/topic/doc_type) и распределение ответов по
  движкам.
- **Граф (LightRAG)** — сущности, связи, чанки, документы (столбцы) и сводные
  показатели (связей на сущность, размер хранилища).
- **Дообучение (LoRA)** — статистика датасета (число пар, средние длины вопроса
  и ответа), параметры LoRA (r, alpha, dropout, число целевых модулей), размер
  адаптера и гистограмма длины ответов.
- **KAG (знание-усиленная генерация)** — карточки (активен ли, декомпозиция, макс.
  шагов, фрагментов на шаг, контекст, знания графа, число ответов через KAG),
  прогресс-бар доли ответов через KAG, кольцевая диаграмма ответов по движкам
  (с долей KAG) и столбцы параметров конвейера (шагов / фрагментов на шаг / контекст),
  схема конвейера и предупреждение, если знания графа включены, но граф не построен.
- **Тайминги по компонентам** — среднее время этапов запроса (поиск/генерация/
  всего), **время обработки (индексация)** — длительность последнего прогона,
  число файлов, среднее на файл и суммарное время, результаты последнего бенчмарка
  по компонентам (мс) и длительность последних операций (индексация, граф,
  дообучение, парсинг сайтов).
- **Время парсинга и эмбеддинга (индексация)** — разбивка времени последней
  индексации на извлечение текста (парсинг/OCR/транскрибация) и эмбеддинг+запись в
  Qdrant: карточки с суммами и долями, кольцевая диаграмма «общий» (парсинг vs
  эмбеддинг), столбчатые «по типам файлов» и «по категориям чанков» (накопительные
  парсинг+эмбеддинг, в секундах), а также «среднее на чанк по типам» (мс/чанк) —
  чтобы видеть, где узкое место (обычно доминирует эмбеддинг, идущий в один поток).
  Данные берутся из `ingest_stats.json` и появляются после индексации (новые
  раздельные тайминги пишутся при следующем прогоне).

### 2.7a. Бенчмарк
Отдельный раздел замера производительности. Кнопка «Запустить бенчмарк» измеряет
скорость каждого компонента и выводит результаты (мс) графиком и карточками + лог:
- **Эмбеддер** — мс/текст и текст/с на пачке;
- **Реранкер** — мс/пара;
- **Поиск + реранк** — среднее время полного retrieval по нескольким запросам;
- **Qdrant** — латентность REST-запроса;
- **LLM генерация** — время и **токены/с** (для Ollama берётся `eval_count`/
  `eval_duration`, для vLLM — `completion_tokens` и время).
Эндпоинт: `POST /api/admin/benchmark`; результаты — в `/api/admin/status`.

### 2.8. Администратор
Управление системой: переключатель режима и движка, операции обслуживания и полный
набор настраиваемых параметров. Защищён токеном (`ADMIN_TOKEN`), если он задан.

**Телефония (SIP/АТС).** Блок **«📞 Телефония (SIP/АТС)»** делает бота телефонным
агентом: Asterisk (или АТС поверх него) по приходу звонка бриджит аудио на наш
сервис через **AudioSocket**, бот распознаёт речь (Whisper), отвечает по базе знаний
(тот же конвейер, что в чате — включая прайс-папку и API-хуки) и озвучивает ответ
(TTS). «Сырую» SIP-регистрацию в приложении не используем — медиамост надёжнее и
портативнее; для АТС без AudioSocket заводится SIP-транк в Asterisk. Настройки —
группа «Телефония (SIP/АТС)» (включение, host:port моста, приветствие, пороги тишины);
в разделе есть статус, кнопка «Перезапустить мост» и готовый пример диалплана
Asterisk. Требуются Asterisk c AudioSocket, ffmpeg, рабочие Whisper и TTS. Эндпоинты:
`GET /api/admin/sip/status`, `POST /api/admin/sip/restart`. Модуль — `sip_bridge.py`.

**Внешние API (хуки).** Блок **«🔌 Внешние API»** — конструктор хуков: для определённых
типов вопросов бот вызывает сторонний REST-сервис и подставляет его ответ в контекст
(модель формулирует ответ по данным со ссылкой на источник). У каждого хука: триггер
(**ключевые слова**, **регэксп** с именованными группами, либо **ИИ-интент**), извлечение
параметров, метод, URL-шаблон с `{параметр}`/`{q}`, заголовки (JSON), тело (для POST),
путь в JSON-ответе (`resp_path`), таймаут. Есть кнопка **«Проверить»** на тестовом
вопросе. Срабатывание видно этапом **🔌 Внешний API** в структуре ответа (чат и
Телеграм). Запросы идут с учётом DNS-записей; короткий кэш по параметрам. Эндпоинты:
`GET /api/admin/api-hooks`, `POST .../api-hooks/{save,delete,test}`; хранилище — таблица
`api_hooks`. Модуль — `api_tools.py`.

**Прайс-папка (контекст без индексации).** На «ценовых» вопросах (цена, стоимость,
прайс, тариф…) ответ можно дополнять содержимым отдельной **папки с прайс-листами
без индексации** в базу: файлы читаются по требованию (кэш по дате изменения),
нарезаются и реранкуются под вопрос; лучшие фрагменты идут в контекст. Удобно для
часто меняющихся прайсов — переиндексация не нужна. Включение, путь и число
фрагментов — настройки `PRICE_FOLDER` / `PRICE_DIR` / `PRICE_TOP_K` (группа «Поиск и
генерация»); статус папки виден в блоке **«💲 Прайс-папка»** раздела «Администратор».
Срабатывание показано этапом **💲 Прайс-папка** в структуре ответа. Модуль —
`price_folder.py`.

**DNS — статические записи.** Блок **«🌐 DNS»** позволяет задать сопоставления
**имя хоста → IP-адрес** прямо в приложении. Это решает ситуацию, когда внутренний
адрес (например `in.vodokomfort.ru`) не резолвится из среды, где запущен RAG (частый
случай в Docker без доступа к внутреннему DNS — ошибка «Не удалось разрешить имя
хоста (DNS)»). Записи применяются ко **всем** сетевым запросам приложения
(синхронизация справочника, парсинг сайтов, Qdrant, LLM и т. п.) через перехват
системного разрешения имён (`dns_override.py`, monkeypatch `socket.getaddrinfo`),
**без перезапуска** и без правки `/etc/hosts`. Имя и IP валидируются; изменения
вступают в силу сразу. Эндпоинты (под `X-Admin-Token`): `GET /api/admin/dns`,
`POST /api/admin/dns/{add,update,delete,clear}`. Хранилище — таблица `dns_hosts`.

## 3. Режимы работы

Вверху раздела «Администратор» — панель «Режим работы». Клик по карточке мгновенно
применяет пресет (нужно ввести токен администратора в поле ниже, если он задан):

| Режим              | Что включает                                                      | Когда использовать |
|--------------------|------------------------------------------------------------------|--------------------|
| **Базовый**        | Вектор + реранк, rule-based авто-фильтр                           | Максимальная скорость, типовые вопросы |
| **С метаданными**  | + умные фильтры из вопроса (категория/продукт/период) через LLM   | Точность на прайсах/каталогах |
| **Hybrid+ (граф)** | + граф-RAG для сводных вопросов (нужен LightRAG)                  | Есть аналитические/сводные вопросы |

Если вручную изменить отдельные флаги, режим отображается как **«Свой»**. Текущий
режим виден в боковом меню на всех вкладках.

### Смена модели LLM

Панель «Модель LLM» в админке показывает текущий бэкенд и список доступных
моделей. Клик по модели переключает генерацию на неё (поле `LLM_MODEL`,
применяется сразу). Для **Ollama** (Apple) можно скачать новую модель: введите
имя (например `qwen2.5:14b-instruct`) и нажмите «Скачать модель» — загрузка идёт
в фоне. В блоке «Состояние и операции» отображается **живой прогресс**: текущий
этап, проценты, скачано/всего и скорость (прогресс берётся из HTTP API Ollama
`/api/pull`; при недоступности — фолбэк на CLI).

**Сохранённые логи индексации.** Полный лог каждой переиндексации и парсинга сайтов
сохраняется в БД (таблица `ingest_logs`) и доступен в блоке «Состояние и операции» →
«Сохранённые логи индексации»: список с временем, типом и сводкой, кнопки «Показать»
(полный текст) и «Удалить»; кнопка «🗑 Удалить все» очищает историю логов. Эндпоинты:
`GET /api/admin/ingest-logs`, `GET /api/admin/ingest-log?id=`,
`POST /api/admin/ingest-logs/delete`.

Для **vLLM** (GPU) у поля `VLLM_MODEL` в настройках есть **выпадающий список
моделей** и кнопка **«🔄 Обновить список»**: список объединяет курируемый каталог
vLLM-моделей и модели, которые сервер vLLM обслуживает прямо сейчас (запрос к
OpenAI-совместимому `/v1/models`). Выбор подставляет имя в `VLLM_MODEL`; затем
«Сохранить настройки» и «Применить модель LLM (рестарт vLLM)». Эндпоинт:
`GET /api/admin/vllm-models`.

### Движок ответов (векторный / LightRAG / KAG)

Над переключателем модели генерации — выбор «Движок ответов»:

- **Векторный (первый вариант)** — поиск + реранк по документам со ссылками на
  источники (рекомендуется по умолчанию).
- **LightRAG (граф)** — ВСЕ вопросы направляются в граф знаний. Требуется
  построенный граф — его можно собрать прямо из админки кнопкой **«Построить граф
  знаний (LightRAG)»** (ставит LightRAG и прогоняет документы через LLM; идёт в
  фоне, статус — в «Состояние и операции»). Если граф недоступен — автоматический
  откат на векторный движок.
- **KAG (знание-усиленная генерация)** — для сложных/составных вопросов. Конвейер:
  декомпозиция вопроса на под-вопросы → мультихоп-поиск по каждому → объединение и
  переоценка фрагментов под исходный вопрос (взаимное индексирование) → опционально
  выжимка из графа знаний → ответ со ссылками. Точнее на многошаговых вопросах
  (сравнения, цепочки, связи), но делает дополнительные вызовы LLM и медленнее.
  Настройки — в группе **«KAG (знание-усиленная генерация)»**.

Отличие от режима «Hybrid+»: там в граф идут только сводные вопросы, а точечные —
через вектор; переключатель движка же отправляет в LightRAG/KAG всё. Управляется
настройкой `ENGINE` (`vector` / `lightrag` / `kag`).

#### Настройки KAG (с рекомендациями)

- **Декомпозиция вопроса** (`KAG_DECOMPOSE`, реком. вкл) и **Макс. под-вопросов**
  (`KAG_MAX_HOPS`, реком. 3) — глубина разбора; 2 для скорости, 4–5 для аналитики.
- **Фрагментов на под-вопрос** (`KAG_CHUNKS_PER_HOP`, 3–5) и **Итоговых фрагментов**
  (`KAG_CONTEXT_CHUNKS`, 6–10) — объём контекста; не превышайте `VLLM_MAX_LEN`.
- **Взаимное индексирование** (`KAG_MUTUAL_INDEX`, реком. вкл) — переоценка
  объединённого пула под исходный вопрос (убирает дубли/шум).
- **Знания графа** (`KAG_GRAPH`) + **Режим графа** (`KAG_GRAPH_MODE`: local/global/…)
  — включайте, когда граф LightRAG построен; без графа KAG работает как мультихоп-RAG.
- **Требовать ссылки** (`KAG_REQUIRE_CITATIONS`, реком. вкл), **Температура**
  (`KAG_TEMPERATURE`, 0.0–0.2), **Системный промпт KAG** (`KAG_SYSTEM_PROMPT`).

### Расширенные настройки OCR (с рекомендациями)

Группа **«OCR (распознавание текста)»** управляет качеством распознавания (tesseract)
для картинок, RAW-фото и «картиночных» страниц PDF (сами флаги «OCR …» — в группе
«Документы и индексация»). Все параметры применяются при переиндексации.

**Описание изображения моделью при пустом OCR.** Если OCR изображения (или
картиночной страницы PDF) дал мало текста — не больше `OCR_LLM_MAX_CHUNKS` чанков
(по умолчанию **1**, т. е. ноль или один), — изображение передаётся **vision-модели**
(`OCR_LLM_DESCRIBE`, `VISION_MODEL`) за описанием, и это описание тоже добавляется в
базу знаний. Полезно для фото, схем, диаграмм, чертежей и сканов без распознаваемого
текста. Нужна мультимодальная модель (например `llava`/`qwen2-vl`/`llama3.2-vision` в
Ollama или vision-модель на vLLM); пустое `VISION_MODEL` = использовать основную LLM.

- **Языки** (`OCR_LANGS`) — через «+», напр. `rus+eng`; пусто = автоопределение.
  Указывайте только нужные языки (лишние замедляют и иногда ухудшают качество).
- **Масштаб рендера PDF** (`OCR_SCALE`, реком. 2.5 ≈ 180 DPI) — выше для мелкого
  шрифта/плохих сканов, ниже для скорости.
- **Порог символов** (`OCR_MIN_CHARS`, реком. 25) — ниже какого объёма текстового
  слоя страница PDF считается «картиночной» и идёт в OCR.
- **PSM** (`OCR_PSM`: 3 авто, 6 единый блок для таблиц/прайсов, 11 разрозненный текст
  для чертежей) и **OEM** (`OCR_OEM`: 3 авто, 1 только LSTM).
- **Макс. сторона** (`OCR_MAX_DIM`, реком. 3500) — даунскейл крупных изображений.
- **Предобработка** (`OCR_PREPROCESS`) — ч/б + автоконтраст; включайте для шумных и
  фотографированных документов.

Подбор: меняйте параметры и переиндексируйте небольшую папку-образец, сверяя
распознанный текст в «Каталоге данных».

### Модель генерации (базовая / дообученная)

Ниже панели режима — переключатель «Модель генерации» с двумя карточками:

- **Базовая модель** — исходная LLM (RAG поверх неё).
- **Дообученная (LoRA)** — модель, дообученная на ваших документах (адаптер в vLLM).

Клик переключает `USE_FINETUNED` мгновенно. Здесь же — **выбор базовой модели для
дообучения**: выпадающий список fp16-моделей, пригодных для QLoRA (Qwen2.5/Qwen3,
Llama 3, Gemma, Mistral, Phi-4), и кнопка **«🔄 Обновить список моделей»**. По
умолчанию база берётся автоматически из `VLLM_MODEL` (с отбрасыванием суффиксов
квантизации `-AWQ/-GPTQ/-Int4`); выбор в списке задаёт её явно (поле
`FINETUNE_BASE`), пункт «из VLLM_MODEL (авто)» возвращает автоопределение.
Эндпоинт: `GET /api/admin/finetune-models`.

Рядом — кнопки **«Запустить дообучение»** (собирает датасет и обучает LoRA-адаптер
на выбранной базе; долго, в фоне) и **«Применить дообученную модель»**
(перезапускает vLLM с адаптером). Карточка «Дообученная» доступна после того, как
адаптер обучен и применён. Дообучение — дополнение к RAG (стиль/терминология),
факты по-прежнему берутся из документов. Подробнее — `finetune/README_finetune.md`.

## 4. Справочник настроек

Все поля редактируются в админке и сгруппированы; у каждого — бейдж, как
применяется изменение: **на лету**, **после переиндексации**, **после рестарта**,
**рестарт vLLM**. Изменения сохраняются в `runtime_config.json`.

### Поиск и генерация (на лету)
- **NO_ANSWER_FALLBACK** (по умолч. выкл.) — если обычный поиск ничего не нашёл (ни один
  фрагмент не прошёл порог), включается расширенный поиск: **лексический** (широкий пул
  кандидатов + совпадение слов запроса с именами файлов) и, если он пуст, **глубокий** —
  LLM по списку имён файлов выбирает 2–3 наиболее вероятных, и ответ ищется в них. Этапы
  фолбэка видны в «Конвейере ответа». Помогает найти документы, которые пропустил
  семантический поиск, ценой дополнительной задержки.

- **TOP_K_RETRIEVE** (5–50, умолч. 20) — сколько кандидатов достаём из Qdrant.
- **TOP_K_RERANK** (1–15, умолч. 6) — сколько фрагментов остаётся после реранка и
  уходит в контекст LLM.
- **MIN_SCORE** (0–1, умолч. 0.30) — порог релевантности после реранка; ниже —
  ответ «не знаю». Выше порог = строже, меньше «притянутых» ответов.
- **TEMPERATURE** (0–1, умолч. 0.1) — креативность генерации; для фактов держите низкой.
- **AUTO_FILTER** (умолч. вкл) — быстрый rule-based фильтр категории по словам вопроса.
- **AUTO_CALIBRATE** (умолч. выкл) — авто-подстройка параметров по оценкам 👍/👎.
- **SYSTEM_PROMPT** — поведение ассистента и правила цитирования.

> В админ-панели под каждым параметром этой группы показано **подробное описание и
> рекомендации по выбору значения** (когда увеличить/уменьшить и на что это влияет).

### Генерация (LLM)
- **LLM_BACKEND** — `ollama` (Apple) или `openai` (vLLM). *После рестарта.*
- **LLM_MODEL** — имя модели для запросов.
- **OLLAMA_URL / LLM_BASE_URL / LLM_API_KEY** — адреса бэкендов.

> Под заголовком этой группы в админке показан блок с описанием бэкендов и
> рекомендациями по выбору модели генерации по объёму памяти и задаче. Полный
> справочник — [docs/MODELS.md](MODELS.md).

### vLLM (GPU)
- **VLLM_MODEL / VLLM_MAX_LEN / VLLM_TP** — модель контейнера, длина контекста,
  число GPU. *Применяются кнопкой «Применить модель LLM».*

> Под заголовком группы — описание моделей vLLM и рекомендации по VRAM (24/48/80 ГБ),
> плюс пояснения к `VLLM_MAX_LEN` и `VLLM_TP`. Полный справочник — [docs/MODELS.md](MODELS.md).

### Эмбеддинги и устройство
- **EMBED_MODEL / RERANK_MODEL** — модели поиска и реранка (после рестарта).
- **EMBED_DIM** — размерность эмбеддингов (после переиндексации). Меняйте вместе с
  EMBED_MODEL: bge-m3, multilingual-e5-large, Qwen3-Embedding-0.6B, jina-v3 — все 1024.
  После смены: «Сбросить индекс» → «Переиндексировать».
- **DEVICE** — `cuda` / `mps` / `cpu`.

> В админ-панели под каждым параметром этой группы выводится **подробное описание и
> рекомендации**: какие модели эмбеддера/реранкера выбирать, какую ставить EMBED_DIM
> и когда нужны «Сбросить индекс»/«Переиндексировать»/«Перезапустить сервис». Полный
> справочник моделей — [docs/MODELS.md](MODELS.md).

### Хранилище
- **QDRANT_URL / QDRANT_COLLECTION** — адрес и коллекция (после рестарта).
- **QDRANT_TIMEOUT** — таймаут запросов из чата, с (после рестарта).
- **QDRANT_INGEST_TIMEOUT** — таймаут операций индексации, с (применяется при
  следующей переиндексации). Повышайте, если на больших файлах бывает «timed out».

### База данных и кэш
- **DB_BACKEND** (`sqlite` по умолчанию | `mysql` | `postgresql`) — где хранятся журнал
  запросов, история Телеграм и аналитика. SQLite — локальный файл, без внешних
  сервисов. MySQL/PostgreSQL — для нескольких инстансов или централизованного
  хранения. Менять напрямую для переезда не нужно — есть кнопки «Мигрировать»
  (см. раздел 5).
- **MYSQL_HOST / MYSQL_PORT / MYSQL_USER / MYSQL_PASSWORD / MYSQL_DB** — подключение к
  MySQL/MariaDB. Базу создайте заранее (`CREATE DATABASE rag CHARACTER SET utf8mb4;`),
  таблицы приложение создаст само. Нужен пакет `PyMySQL`.
- **PG_HOST / PG_PORT / PG_USER / PG_PASSWORD / PG_DB** — подключение к PostgreSQL.
  Базу создайте заранее (`CREATE DATABASE rag;`). Нужен пакет `psycopg2`.
- **REDIS_ENABLED** (по умолчанию выкл.) + **REDIS_HOST / REDIS_PORT / REDIS_DB /
  REDIS_PASSWORD** — опциональный кэш: эмбеддинг запроса, результаты поиска, агрегаты
  (статистика/аналитика/Телеграм), тяжёлые админские чтения. Инвалидация раздельная
  (агрегаты — при записи в журнал; поиск/ответы/каталог — при переиндексации). Без Redis
  всё работает напрямую. Нужен пакет `redis`.
- **ANSWER_CACHE** (по умолчанию выкл., требует Redis) — кэшировать готовые ответы LLM:
  одинаковый вопрос отдаётся из кэша без повторного вызова модели (до переиндексации или
  смены модели/промпта/температуры).

### Документы и индексация (после переиндексации)
- **DOCS_DIR** — папка с документами. Рядом с полем — кнопка **«Обзор…»**: открывает
  браузер папок на сервере (навигация по подпапкам и вверх, счётчик документов в
  папке). Выбрав папку, нажмите «Сохранить настройки», затем «Переиндексировать».
- **CHUNK_SIZE** (умолч. 900) / **CHUNK_OVERLAP** (умолч. 150) — нарезка текста.
- **OCR_IMAGES / OCR_RAW / PARSE_CAD / TRANSCRIBE_AV** — включать ли тяжёлые
  экстракторы: OCR картинок, OCR RAW-фото, чтение чертежей CAD, транскрибацию
  аудио/видео. Это самые медленные этапы. Для **ускорения индексации** большого
  каталога отключите их, сделайте быструю первую переиндексацию (PDF/DOCX/таблицы/
  текст проиндексируются за минуты), затем включите нужные и переиндексируйте ещё
  раз — благодаря инкрементальности добавятся только новые типы.
- **INGEST_WORKERS** (умолч. 0 = авто) — сколько файлов парсить **параллельно** при
  индексации. Самое долгое — извлечение (парсинг/OCR/конвертация DWG/транскрибация);
  при последовательной обработке сервер часто недозагружен (ядра простаивают, пока один
  файл парсится). Несколько потоков извлечения держат конвейер занятым и ускоряют
  индексацию в разы. 0 — авто (по числу ядер, ≤8); рекомендация 4–8. Эмбеддинг и запись
  в Qdrant идут в основном потоке. ⚠️ При заданном `FILE_PARSE_TIMEOUT` параллельность
  отключается (лимит времени на файл работает только однопоточно).
- **EMBED_BATCH** (умолч. 32) — размер пачки эмбеддера: сколько чанков считать за один
  проход. Эмбеддинг идёт в **один поток** и часто и есть главный тормоз индексации (число
  «потоков извлечения» на него не влияет). На GPU чанки в пачке считаются параллельно,
  поэтому больший batch повышает пропускную способность. Рекомендации: CPU — 16–32;
  Apple MPS — 32–64; CUDA 12–16 ГБ — 64; 24 ГБ и больше — 128+. Плата — видеопамять:
  при ошибке нехватки памяти (OOM) уменьшите. Применяется при следующей индексации.
- **FILE_PARSE_TIMEOUT** (умолч. 0) — лимит времени на один файл, с; превышение —
  файл пропускается (защита от «зависания» на тяжёлом DWG/видео). Работает на
  Linux/macOS. Пропущенные по таймауту видны в логе и сводке индексации. ⚠️ Включает
  однопоточный режим (отключает параллельное извлечение).

> Скорость индексации: при `21 с/файл` и тысячах файлов узкие места — это OCR
> картинок (~секунды на файл), конвертация DWG и транскрибация видео (минуты на
> файл). Отключение этих экстракторов на первый проход сокращает время в разы.
> Предупреждение openpyxl «Data Validation extension is not supported» —
> безвредное, теперь подавляется.

### Транскрибация (после переиндексации)
- **WHISPER_BACKEND** (`faster`/`mlx`) / **WHISPER_MODEL**.

### Расширенный поиск (hybrid+)

### Дообучение (fine-tuning)

- **FINETUNE_BASE** — базовая fp16-модель для дообучения. Пусто = берётся из

> Под каждым параметром этой группы в админке выводится **подробное описание и

## 4a. Добавление файлов через веб-интерфейс

В админке есть блок **«Добавить файлы (Excel и др.)»**: выберите один или несколько
файлов и нажмите «Загрузить» — они сохранятся в подпапку `uploads` папки
документов. Кнопка «Загрузить и переиндексировать» сразу запустит индексацию.
Принимается широкий набор форматов: PDF, DOCX/DOC, PPTX, XLSX/XLSM/XLS/CSV,
TXT/MD, HTML/MHTML, XML/JSON, SVG, ярлыки `.url`, письма Outlook `.msg`, чертежи
DXF/DWG и 3D-CAD STEP/IGES, изображения (JPG/PNG/…) и RAW-фото (CR2/…) с OCR, а
также архивы (ZIP/RAR/7Z/TAR — распаковываются, содержимое индексируется).
Excel/CSV разбираются построчно в формат «колонка: значение». Файлы
неподдерживаемых типов пропускаются с пометкой.

**Загрузка каталога целой папкой.** В том же блоке — выбор папки (кнопка выбора
каталога) и «📁 Загрузить папку». Структура подпапок сохраняется в папке
документов. Рассчитано на очень большой каталог (десятки тысяч файлов, проверено до
~50 000): загрузка идёт **порциями по 200 файлов** с индикатором прогресса и
кнопкой «Стоп»; неподдерживаемые типы отсеиваются ещё в браузере, чтобы не гонять
лишнее. Эндпоинт: `POST /api/admin/upload-folder` (files[] + paths[]). Опасные пути
(`..`) обезвреживаются, запись только внутрь папки документов.

## 4b. Сайты в базе знаний

Блок «Сайты в базе знаний» в админке: добавляйте адреса **по одному** — введите URL и
нажмите «➕ Добавить» (или Enter); адрес попадает в **очередь** в виде элемента, который
можно убрать крестиком (всего до 50). Кнопка «Очистить очередь» убирает все. Затем нажмите
«🌐 Спарсить и добавить в базу» (адрес, оставшийся в поле ввода, подхватывается
автоматически). Схема (`https://`) подставляется, если не указана. С каждого адреса выполняется **обход ссылок**
в пределах того же домена. Со страниц извлекается **основной текст** (через
`trafilatura`, с запасным разбором BeautifulSoup — отбрасываются меню/скрипты/стили,
сохраняются таблицы) и объединяется в один документ в подпапке `web/`. Дополнительно
**скачиваются все файлы по ссылкам** (PDF, изображения, документы, архивы, медиа — без
фильтра по типу) в подпапку `web/files/` и индексируются обычными загрузчиками. Список
адресов запоминается.

**Headless-браузер для JS-сайтов.** Если сайт рендерится через JavaScript (контент
подгружается скриптами, в исходном HTML текста нет), включён **Playwright Chromium**: он
открывает страницу как настоящий браузер и отдаёт уже отрисованный HTML. В Docker браузер
ставится при сборке образа; при ручной установке выполните
`pip install playwright && playwright install chromium`. Если браузер недоступен —
автоматический откат на обычную загрузку (в логе будет пометка).

Параметры — в группе настроек **«Сайты (парсинг)»**:
- **WEB_CRAWL_DEPTH** (по умолч. 1) — глубина обхода ссылок: 0 — только указанная
  страница; 1 — она и страницы, на которые она ссылается; 2 — ещё глубже. Файлы по ссылкам
  скачиваются на любой глубине, в т.ч. при 0.
- **WEB_MAX_PAGES** (по умолч. 20) — максимум страниц с одного сайта.
- **WEB_MAX_FILES** (по умолч. 5000) — максимум скачиваемых файлов с одного сайта (по
  количеству; типы не фильтруются).
- **WEB_JS_RENDER** (по умолч. вкл.) — рендерить страницы headless-браузером.
- **WEB_SAME_DOMAIN** (по умолч. вкл.) — не уходить на внешние домены.

Прямо в этом блоке:
- **Список «Сайты в базе»** — все добавленные сайты; у каждого кнопка **«Удалить»**,
  которая убирает сайт из списка, удаляет его файл из папки `web/` и его чанки из
  базы знаний (Qdrant) — без полной переиндексации.
- **Лог парсинга** — подробный живой лог: по каждому сайту видно обходимые страницы
  (`стр. k/max`), список и счётчик скачиваемых файлов (`[файл i/N] %`) с **процентом
  загрузки текущего файла**, затем ход индексации — по каждому файлу строка
  `[i/всего] % индексирую: <файл>`, а для крупных файлов ещё и процент по самому файлу
  (`<файл>: % (чанков done/всего)`). Обновляется автоматически во время работы.

## 4c. Удалённые хосты (Qdrant / LightRAG по SSH)

Блок «Удалённые хосты» позволяет разворачивать и использовать компоненты на других
серверах. Профиль хоста: имя, ОС (Linux/macOS/Windows), адрес, SSH-порт, логин,
пароль, опционально `Qdrant URL` / `vLLM URL` / папка документов. Кнопки у каждого
хоста:

- **Развернуть Qdrant** — по SSH ставит/запускает контейнер Qdrant на хосте.
- **Развернуть LightRAG** — создаёт venv и ставит зависимости LightRAG на хосте.
- **Данные → / ←** — перенос папки документов на хост и обратно (SFTP), затем
  можно переиндексировать на месте.
- **Snapshot → / ←** — перенос готового индекса Qdrant между локальным и удалённым
  (через snapshot-API Qdrant).
- **Переключиться** — переводит приложение на удалённые `Qdrant URL`/`vLLM URL`
  (с сохранением локальной конфигурации); кнопка **«Вернуться на локальный»** —
  обратно. После переключения перезапустите сервис.

Ход операций виден в логе под списком. Ограничения и безопасность: SSH по
логину/паролю (нужна библиотека `paramiko`); на Linux требуется root или
passwordless sudo для установки Docker; на macOS/Windows Docker должен быть
предустановлен, а SSH-доступ включён; пароли хранятся в `remote_hosts.json` —
держите хосты в доверенной сети. Windows-деплой — best-effort (зависит от
настроек OpenSSH/PowerShell и Docker Desktop).

## 5. Операции обслуживания (админка)

Блок «Состояние и операции» показывает доступность Qdrant и LLM, число чанков,
статус задач и **живой лог** текущей операции (индексация, граф, дообучение,
установка) — он обновляется по ходу выполнения, а в конце показывает сводку
(`SUMMARY ... errors=N`). Кнопки:

- **Переиндексировать** — фоновый запуск `ingest.py`; инкрементально обновляет
  только изменённые файлы. Нужно после смены `DOCS_DIR`, чанкинга, включения
  LLM-метаданных или добавления документов.
- **Переиндексировать с нуля** — пересоздаёт коллекцию (`--reset`).
- **Применить модель LLM (рестарт vLLM)** — записывает `VLLM_*` и перезапускает
  контейнер vLLM с новой моделью (GPU).
- **Перезапустить сервис** — корректно завершает API; systemd (`Restart=always`)
  поднимает его заново. Нужно для параметров со scope «после рестарта».
- **Установить Qdrant + зависимости** — поднимает контейнер Qdrant (нужен Docker).
- **Установить LightRAG + зависимости** — pip-установка LightRAG в окружение (без
  построения графа).
- **Построить граф знаний** — установка LightRAG + `graph_rag ingest` (фоном);
  заменяет ручной запуск `gpu_variant/setup_hybrid.sh`.
- **Запустить дообучение** — фоновый пайплайн QLoRA (датасет + обучение адаптера).
- **Применить дообученную модель** — перезапуск vLLM с LoRA-адаптером.

### Калибровка по оценкам ответов

Блок «Калибровка по оценкам ответов» в админке использует оценки 👍/👎 из чата:
- **Проанализировать оценки** — показывает статистику (хорошие/плохие, средняя
  релевантность, плохие «без ответа» и «с ответом») и рекомендации по настройкам
  (порог `MIN_SCORE`, объём выборки `TOP_K_*`, `TEMPERATURE`, умные фильтры).
- **Применить рекомендации** — применяет предложенные изменения (с сохранением
  предыдущих значений для отката).
- **Откатить изменения** — возвращает настройки к значениям до последней калибровки.
- **Галка «Автоматически калибровать»** (`AUTO_CALIBRATE`) — каждые 10 новых оценок
  система сама применяет рекомендации.

Нужно ≥ 5 оценок для анализа. Логика эвристическая: частые «не знаю» при плохих
оценках → ослабить порог и расширить выборку; частые плохие при наличии ответа →
снизить температуру, включить умные фильтры, поднять порог.

### Автокалибровка параметров на тестовом наборе

Отдельный раздел навигации **«🎛 Автокалибровка»** подбирает `MIN_SCORE`,
`TOP_K_RERANK` и `TOP_K_RETRIEVE` на эталонном наборе «вопрос → ожидаемый результат»
(в отличие от калибровки по живым оценкам — здесь набор задаёте вы). Раздел содержит:
редактор тестового набора, запуск, **анимированную схему процесса** (этапы Набор →
Поиск+реранк → Перебор сетки → LLM-проверка → Отчёт с подсветкой текущего этапа),
**живые метрики и графики** (лучшая точность по ходу перебора, прогресс этапа,
карточки с числом вопросов/комбинаций/временем) и **финальный отчёт с графиками**
(точность по каждому параметру, текущие vs рекомендованные, топ-комбинации) и
пояснениями к каждому элементу.

Во время работы доступны **кнопка «⏹ Остановить»** (мягкая отмена калибровки/оценки) и
**подробный живой лог** (этапы, лучшие комбинации, повопросные результаты) с кнопкой
«💾 Сохранить лог в БД» (запись в `ingest_logs` с возможностью удаления). В **оценке по
набору из папки `test`** дополнительно строится **интерактивный график сходимости**:
сходство ответа системы с эталоном по каждому вопросу, накопленная доля зачётов и линия
порога — в реальном времени. Эндпоинты: `POST /api/admin/calib/cancel`,
`POST /api/admin/calib/auto/cancel`, `POST /api/admin/calib/save-log` (`{which:'calib'|'auto'}`).

**Тестовый набор** редактируется **графическим конструктором** в разделе
«🎛 Автокалибровка» (карточки вопросов: поле вопроса + теги ожидаемых файлов-источников
и слов ответа, добавляются по Enter; тип «позитивный/негативный» подсвечивается
автоматически). Кнопки: «➕ Добавить вопрос», «Загрузить», «💾 Сохранить», «Вставить
пример», «Очистить». Для опытных есть сворачиваемый раздел **JSON** с двусторонней
синхронизацией («Обновить JSON из конструктора» / «Применить JSON в конструктор»).
Набор хранится в БД (`kv_store`, ключ `calib_testset`; переносится при копировании/
миграции между СУБД). Каждый элемент:

```json
{"q": "вопрос",
 "sources": ["часть имени ожидаемого файла"],
 "answer":  ["ключевое слово из ответа"]}
```

- `sources` — попадание засчитывается, если среди выбранных фрагментов есть файл,
  чьё имя содержит любую из строк.
- `answer` — попадание засчитывается, если в тексте выбранных фрагментов есть любое
  из ключевых слов; при включённой галке «проверять и текст ответа (LLM)» для
  рекомендованной и текущей комбинаций дополнительно генерируется ответ LLM и
  проверяется наличие этих слов в самом ответе.
- Элемент **без** `sources` и `answer` — **негативный**: правильный результат —
  честное «не знаю» (после порога не остаётся ни одного фрагмента). Добавляйте
  такие вопросы, чтобы система не «галлюцинировала» на отсутствующих темах.

*Сохранённые наборы вопросов.* Блок «Сохранённые наборы вопросов»: задайте имя и нажмите
«💾 Сохранить набор» — в БД (таблица `calib_sets`, `kind='manual'`) сохраняются текущие
вопросы конструктора и **найденные параметры** последней калибровки (лучшие
`MIN_SCORE`/`TOP_K` по режимам, с точностью). Список показывает имя, дату, число вопросов и
параметры; набор можно **загрузить** обратно в конструктор или **удалить**. Эндпоинты:
`GET /api/admin/calib/sets`, `POST /api/admin/calib/sets/save|load|delete`.

**Альтернатива — набор из папки `test` (авто Q&A).** Отдельный блок: положите
файлы-образцы в подпапку **`test`** каталога документов, нажмите «🧪 Сгенерировать из
папки test» — LLM создаст пары «вопрос → эталонный ответ» (по умолчанию 50), которые
выводятся таблицей (вопрос, эталон, источник) и хранятся в БД (`kv_store`,
`calib_testset_auto`). Кнопка «▶ Прогнать оценку» прогоняет каждый вопрос через текущий
или выбранный движок, и ответ системы сравнивается с эталоном по **смысловой близости**
(косинус эмбеддингов). Поле **«Допустимое отклонение ответа, %»** задаёт порог: отклонение
= (1 − близость)·100 %, зачёт — если отклонение ≤ порога (рекоменд. 20–35 %). Результат —
точность и таблица по парам (вопрос, эталон, ответ системы, сходство %, отклонение %,
вердикт) + полный отчёт в БД с возможностью удаления. Это оценка качества ответа целиком,
дополняющая подбор порога/выборки.

*Редактируемый промпт генерации.* В сворачиваемом блоке «✏️ Промпт генерации (LLM)» можно
посмотреть и изменить системный промпт, по которому LLM строит пары (он должен возвращать
JSON `{"q":…,"a":…}`), сохранить его или вернуть значение по умолчанию. Промпт хранится в БД
(`kv_store`) и используется при генерации.

*LLM-автокалибровка (итеративная, универсальная).* Блок «🤖 LLM-автокалибровка»: система
сама подбирает параметры в цикле. Режим **универсален для любого варианта работы** —
выберите **движок** (Векторный/LightRAG/KAG) и **режим** (Базовый/С метаданными/Hybrid+);
оптимизатор активирует этот вариант и подбирает **только актуальные для него параметры**
(определяются автоматически): для вектора — `MIN_SCORE`, `TOP_K_RETRIEVE`, `TOP_K_RERANK`,
`TEMPERATURE`, `NO_ANSWER_FALLBACK`, фильтры `AUTO_FILTER`/`SMART_FILTER`; для LightRAG —
`GRAPH_MODE` (+`TEMPERATURE`); для Hybrid+ — ещё `GRAPH_RAG`/`GRAPH_MODE`; для KAG — его
параметры (`KAG_MAX_HOPS`, `KAG_CHUNKS_PER_HOP`, `KAG_CONTEXT_CHUNKS`, `KAG_TEMPERATURE`,
`KAG_GRAPH`, `KAG_MUTUAL_INDEX`, `KAG_GRAPH_MODE`).

На каждой итерации: (1) набор оценивается текущими параметрами (ответ сравнивается с эталоном
по сходству); (2) LLM получает подробный контекст — структуру проекта и взаимодействие
компонентов, активный вариант, актуальные параметры с диапазонами/вариантами и их смыслом,
пары вопрос/эталон, тестовые файлы и результаты предыдущей итерации (в т.ч. провалившиеся
вопросы); (3) LLM возвращает рекомендуемые значения (числа, true/false, варианты), они
проверяются по типам/границам и применяются; (4) запускается следующая итерация. Цикл идёт
до **лимита циклов** (редактируемый, по умолчанию 200) или до сходимости (100% или
стабилизация параметров); в конце применяется лучший набор. Во время прогона —
**график сходимости в реальном времени**, **кнопка «⏹ Остановить»** и **живой лог** с
сохранением в БД. ⚠️ Каждая итерация вызывает LLM на все пары плюс запрос параметров —
долго и затратно; начинайте с небольшого лимита. Эндпоинты:
`GET /api/admin/calib/opt/variants`, `POST /api/admin/calib/opt/run`
(`{max_iter, deviation, engine, mode}`), `GET /api/admin/calib/opt/status`,
`POST /api/admin/calib/opt/cancel`.

*Сохранённые наборы (промпт + пары + параметры).* Блок «Сохранённые наборы»: задайте имя и
нажмите «💾 Сохранить набор» — в БД (таблица `calib_sets`) сохраняются текущий промпт,
сгенерированные пары и параметры последнего прогона (движок, допуск, точность,
MIN_SCORE/TOP_K). Список наборов показывает имя, дату, число пар и параметры; набор можно
**загрузить** обратно (восстанавливает промпт и пары) или **удалить**.

Эндпоинты: `GET /api/admin/calib/auto/testset`,
`POST /api/admin/calib/auto/generate` (`{n, folder, prompt}`), `GET /api/admin/calib/auto/status`,
`POST /api/admin/calib/auto/run` (`{deviation, engine}`),
`GET/POST /api/admin/calib/auto/prompt`, `GET /api/admin/calib/auto/sets`,
`POST /api/admin/calib/auto/sets/save|load|delete`. Каждый прогон/генерация вызывают LLM на
все пары — это занимает время.

**Калибровка по режимам работы (вариантам).** В блоке «Запуск» можно выбрать один или
несколько режимов работы (Базовый / С метаданными / Hybrid+) — они отличаются фильтрацией
кандидатов перед реранком, поэтому оптимальные `MIN_SCORE`/`TOP_K_*` у них разные.
Параметры подбираются под каждый выбранный режим отдельно; при выборе нескольких калибровка
идёт по очереди. В результатах для каждого режима — свой блок с карточками, графиками и
таблицей, а при нескольких режимах сверху появляется сводная таблица сравнения. Применить
можно по каждому режиму: «Применить рекомендованные» (только параметры) или «Применить и
включить режим» (параметры + переключение активного режима). По умолчанию отмечен текущий
активный режим. Кнопки **«Выбрать все»/«Снять все»** калибруют сразу все режимы. Граф-RAG
(Hybrid+) для сводных вопросов не параметризуется этими настройками — калибруется его
векторная часть.

**Оценка движков (vector / LightRAG / KAG).** Отдельный блок «Движки (оценка end-to-end)»:
выбранные движки прогоняются по всему тестовому набору на **текущих** настройках, и
измеряется их точность/полнота/специфичность — видно, какой вариант работы лучше на ваших
данных. В результатах — сравнительная диаграмма и таблица; кнопка **«Сделать активным»**
переключает движок (`ENGINE`). В отличие от калибровки режимов (подбор порога/выборки сеткой,
без LLM), оценка движков — это сквозной прогон: KAG и LightRAG вызывают LLM на каждый вопрос,
поэтому идёт дольше. Внутренние параметры KAG и графа подбираются вручную в их группах
настроек. Можно запустить **только** оценку движков (без режимов) или вместе с калибровкой
режимов.

**Конфигурируемая сетка перебора (по компонентам).** В блоке «Запуск» — редактор
**«Параметры перебора»**, сгруппированный по компонентам (Поиск, Реранк, KAG, Индексация). Для
каждого параметра — подробное описание и рекомендации. У сеточных параметров (`TOP_K_RETRIEVE`,
`MIN_SCORE`, `TOP_K_RERANK`) есть **галочка включения** и поля **мин / макс / шаг**: отмеченные
варьируются по диапазону, снятые берутся из текущих настроек одним значением; рядом показывается
число значений и **итоговое число комбинаций**. Кнопка «Сбросить к умолчаниям» возвращает
диапазоны. Параметры KAG и индексации/эмбеддинга показаны информационно (текущие значения +
рекомендации) — они не входят в быстрый перебор: KAG оценивается end-to-end через «Движки», а
чанкинг/эмбеддинг требуют переиндексации.

По завершении **полный расширенный отчёт** (пояснения метрик, все режимы и комбинации,
повопросный разбор при рекомендованных параметрах с выбранными фрагментами и
вердиктами, при необходимости — LLM-проверка) автоматически сохраняется в БД
(таблица `ingest_logs`, метка «Автокалибровка»). Его можно открыть кнопкой
«📄 Полный лог» прямо в результатах и удалить кнопкой «🗑 Удалить сохранённый лог»
или из блока «Сохранённые логи индексации и калибровки».

Кнопки: «Загрузить», «💾 Сохранить», «Вставить пример», «▶️ Запустить калибровку».
Поиск и реранк по каждому вопросу выполняются **один раз** (по широкому пулу), затем
перебирается сетка параметров — это лишь нарезка уже посчитанных кандидатов, поэтому
быстро. Метрика — **точность** (доля корректных: нашёл нужное / корректно промолчал),
дополнительно показываются **полнота** (по позитивным) и **специфичность** (по
негативным). При равной точности предпочитается более высокий порог (честнее) и
меньшая выборка. Таблица показывает топ комбинаций, рекомендованная подсвечена;
кнопка **«Применить рекомендованные»** записывает параметры в настройки и сбрасывает
кэш поиска.

Эндпоинты: `GET/POST /api/admin/calib/testset`, `GET /api/admin/calib/example`,
`POST /api/admin/calib/run` (`{use_llm, grid}`), `GET /api/admin/calib/status`,
`POST /api/admin/calib/apply`.

### Резервное копирование и восстановление

Блок «Резервное копирование» в админке. Три области копирования:

- **Копия настроек** (`settings`) — только конфигурация: `runtime_config.json`, `.env`.
- **Копия служебных данных без каталога** (`service`) — настройки + операционные
  данные: журнал/оценки (`rag_logs.db`), тайминги (`ingest_stats.json`), SSH-хосты
  (`remote_hosts.json`), граф знаний (`graph_storage/`), LoRA-адаптер и датасет
  (`finetune/adapter`, `finetune/data`). Папка документов НЕ включается.
- **Полная копия** (`full`) — всё перечисленное + каталог документов (`DOCS_DIR`).

Каждый архив — `tar.gz` с `manifest.json`, где для каждого файла записан SHA-256;
рядом кладётся `<архив>.sha256`. **Контроль целостности** выполняется автоматически
сразу после создания (пересчёт хешей) и виден в списке. Архивы лежат в `backups/`,
их можно скачать (⬇️) или удалить (🗑). Векторный индекс Qdrant в копию не входит
(он производный) — после восстановления данных выполните переиндексацию.

**Восстановление**: загрузите архив, нажмите **«Проверить архив»** — выполняется
предварительный контроль целостности и состава (область, число файлов, проблемы);
кнопка «Восстановить» становится активной только если архив корректен. При
восстановлении сначала создаётся **страховочная копия** текущих служебных данных,
затем файлы распаковываются строго внутрь каталога проекта/`DOCS_DIR` (защита от
обхода путей). Эндпоинты: `POST /api/admin/backup/create|verify|restore|delete`,
`GET /api/admin/backup/list|download`. Создание и восстановление идут в фоне (статус
обновляется в блоке).

### База данных и кэш (SQLite / MySQL / PostgreSQL / Redis)

Блок «🗄 База данных и кэш» в админке. По умолчанию данные (журнал запросов, история
Телеграм, аналитика) хранятся в локальном **SQLite** (`rag_logs.db`) — без внешних
сервисов. При желании можно перенести их в **MySQL/MariaDB** или **PostgreSQL** —
например, чтобы несколько инстансов смотрели в общую БД.

Порядок переезда:

1. Создайте пустую базу на сервере СУБД (`CREATE DATABASE rag CHARACTER SET utf8mb4;`
   для MySQL или `CREATE DATABASE rag;` для PostgreSQL). Таблицы приложение создаст само.
2. В группе настроек **«База данных и кэш»** заполните хост/порт/логин/пароль/имя БД,
   нажмите **«Сохранить настройки»**.
3. **«Проверить MySQL/PostgreSQL»** — убедитесь, что подключение успешно.
4. Кнопки **«Копировать данные и настройки»** — копируют данные текущей активной БД в
   выбранную (таблицы приёмника очищаются — полная замена); активный бэкенд не
   меняется. Настройки (`runtime_config.json`) при этом сохраняются в таблицу
   `kv_store` приёмника (для переносимости между хостами).
5. Кнопки **«Мигрировать»** — копируют данные **и** переключают приложение на выбранную
   СУБД. После миграции выполните **«Перезапустить сервис»**. «Мигрировать обратно на
   SQLite» возвращает хранение в локальный файл.

Карточки статуса показывают для каждого бэкенда: настроен ли он, есть ли драйвер
(`PyMySQL` / `psycopg2`), доступен ли сервер, и какой бэкенд сейчас активен.
Записи в журнал устойчивы к сбою внешней БД — при недоступности запись пропускается,
а чат продолжает работать.

**Каталог документов в PostgreSQL.** Если активная БД — **PostgreSQL**, в блоке
появляется подраздел «Каталог данных в PostgreSQL»:

- **«Загрузить каталог данных в PostgreSQL»** — переносит документы **целиком**:
  содержимое файла (`content`), имя файла, расширение, размер, дату, способ извлечения,
  а также **контрольную сумму SHA-256** и текст для предпросмотра (берётся из готового
  индекса Qdrant). Цель — загрузить один раз и больше не обращаться к папке с данными.
  Небольшие файлы пишутся пакетами (bytea), **крупные файлы (от 64 МБ) в PostgreSQL —
  как Large Object** потоково, без загрузки в память: это обходит лимит `bytea` в 1 ГБ
  и позволяет хранить файлы **до 100 ГБ**. **Повторная загрузка** пропускает неизменённые
  файлы: сначала по размеру и дате, затем по SHA-256 — поэтому второй запуск быстрый и
  грузит только новое/изменённое. Идёт в фоне с прогресс-баром (сохранено / без изменений /
  ошибок). Файлы больше 100 ГБ сохраняются без содержимого (только метаданные). Совет:
  сначала проиндексируйте каталог, чтобы был текст для предпросмотра.
- **«Очистить каталог данных в PostgreSQL (только файлы)»** — удаляет из БД **только
  содержимое файлов** (и контрольные суммы), оставляя метаданные и текст. Освобождает
  место; при следующей загрузке файлы сохранятся заново.
- **«Перейти на работу с каталогом данных в PostgreSQL»** (активна после загрузки) —
  переключает источник каталога: список файлов в «Каталоге документов» и просмотр
  текста файла начинают читаться из PostgreSQL, а не из папки. Удобно, если папка с
  документами не смонтирована на этом хосте.
- **«Вернуться на работу с каталогом данных»** — возвращает источник на файловую
  систему (`DOCS_DIR`).

Текущий источник виден в карточке и в сводке «Каталога документов» («Источник
каталога»). Управляется настройкой `CATALOG_SOURCE` (`filesystem`/`postgresql`).

**Полностью без папки с файлами (хранение только в PostgreSQL).** Сценарий «загрузить
один раз и работать только с базой»:

1. Скопируйте файлы в папку `DOCS_DIR`.
2. **«Загрузить каталог данных в PostgreSQL»** — файлы (целиком) попадают в `doc_catalog`.
3. **«Перейти на работу с каталогом данных в PostgreSQL»** (`CATALOG_SOURCE=postgresql`).
4. **«Переиндексировать»** (блок «Состояние и операции») — теперь `ingest.py` берёт
   документы **из PostgreSQL** (а не из папки): для каждого файла содержимое временно
   распаковывается, извлекается текст, строится индекс Qdrant, а извлечённый текст
   сохраняется обратно в `doc_catalog` для предпросмотра.

После этого папку с документами можно удалить: поиск и ответы работают из Qdrant,
список каталога и просмотр текста — из PostgreSQL, а **медиа и исходники файлов**
(скачивание файла, превью изображений/PDF/чертежей, кадры и фрагменты видео) отдаются
из PostgreSQL: при первом обращении содержимое выгружается из `doc_catalog` во
временный кэш (`previews/`) и обрабатывается ffmpeg/Pillow как обычный файл. Повторная
индексация и добавление новых файлов тоже идут из PostgreSQL.

Когда источник каталога = `postgresql` (и активна PostgreSQL), **новые документы сразу
попадают в PostgreSQL**: файлы и папки, загружаемые через веб-интерфейс («Добавить
файлы», «Загрузить папку»), и страницы, полученные парсингом сайтов, кроме записи на
диск также целиком сохраняются в `doc_catalog` (с SHA-256). Поэтому последующая
переиндексация из БД их учитывает, и папка для этого не нужна.

**Кэш Redis** (опционально, по умолчанию выключен): флажок `REDIS_ENABLED` + параметры
подключения. Без Redis всё работает напрямую; кнопка **«Очистить кэш Redis»** сбрасывает
кэш вручную. Что кэшируется:

- **эмбеддинг запроса** — вектор вопроса (ключ привязан к модели эмбеддингов); повторные
  вопросы не кодируются заново;
- **результаты поиска** — выдача `search()` (Qdrant + BM25 + реранк) для одинаковых
  вопросов и настроек; пропускает весь конвейер поиска;
- **готовые ответы LLM** — опционально, флажок **`ANSWER_CACHE`** (по умолчанию выкл.):
  одинаковый вопрос отдаётся из кэша без вызова LLM (учитываются промпт, модель,
  температура, фильтры; история диалога — только без неё);
- **тяжёлые админские чтения** — фасет каталога по Qdrant и сводка раздела «Система»
  (короткий TTL);
- **агрегаты** — статистика дашборда, аналитика и статистика Телеграм.

**Инвалидация раздельная (по пространствам):** агрегаты сбрасываются при каждой новой
записи в журнал; кэш поиска/ответов/каталога — при **переиндексации** или сбросе индекса
(а также автоматически при смене влияющих настроек — они входят в ключ). Поэтому данные
не устаревают, а попадания кэша становятся осмысленными.

Драйверы (`PyMySQL`, `psycopg2-binary`, `redis`) входят в `requirements*.txt`; при
ручной установке поставьте их через «Переустановить окружение» или `pip install`.
Эндпоинты: `GET /api/admin/db/status`, `POST /api/admin/db/test|copy|migrate`,
`POST /api/admin/cache/clear`. Копирование/миграция идут в фоне (статус — в блоке).

### Переустановка и сброс

Блок «Переустановка и сборос» в админке:
- **Сброс данных** — кнопки очистки индекса Qdrant, графа, LoRA-адаптера и датасета,
  журнала, настроек, либо «Сбросить всё». `.env` и сам сервис сохраняются.
- **Переустановить окружение** — пересоздаёт `.venv` и переустанавливает
  зависимости, затем перезапускает сервис (фоновый `reinstall.sh`).
- **Полная переустановка** (DESTRUCTIVE) — кнопки «Полная переустановка
  (GPU-сервер)» и «Полная переустановка (Mac)»: останавливают сервис и контейнеры,
  удаляют окружение и ВСЕ данные (индекс, граф, адаптер, журнал) и разворачивают
  заново; `.env` сохраняется, сервис недоступен во время процесса. Те же действия
  доступны из shell: `sudo bash reinstall_server.sh` / `bash mac_variant/reinstall_mac.sh`
  (в скриптах можно пропустить вопрос подтверждения переменной `CONFIRM=yes`).
  Для GPU полной переустановке нужны права root (apt/systemd/docker).

## 6. Регулярное обновление индекса

Чтобы новые/изменённые файлы попадали в базу автоматически:

- **Apple**: пример задачи launchd — `reindex.plist` (каждый час). Установка
  описана в комментариях файла.
- **GPU/Linux**: cron, например ежечасно:
  ```
  0 * * * * cd /opt/rag && ./.venv/bin/python ingest.py >> /var/log/rag_reindex.log 2>&1
  ```

Индексация инкрементальна (хеш + mtime), поэтому повторные запуски дёшевы.

## 7. HTTP API

Базовый URL: `http://<сервер>:8000`. Админские эндпоинты требуют заголовок
`X-Admin-Token`, если задан `ADMIN_TOKEN`.

### Чат
`POST /chat` — тело `{"question": str, "history": [{role,content}], "filters": {...}|null}`.
Ответ — поток NDJSON: строки `{"type":"answer","text":...}` и финальная
`{"type":"sources","items":[...]}`. Поле `filters` (необязательно) задаёт явные
условия Qdrant, например `{"doc_category":"price"}`.

### Чат с приложенным документом
`POST /chat-doc` (multipart): `file`, `question`, `history` (JSON-строка). Парсит
файл, отбирает релевантные фрагменты и отвечает по ним; ответ — поток NDJSON, как
у `/chat`. Без индексации в Qdrant.

### Оценки и калибровка
- `POST /api/rate` — оценка ответа `{"id":N,"rating":1|-1|0}` (без токена, из чата).
- `GET /api/admin/recommendations` — анализ оценок и рекомендации.
- `POST /api/admin/apply-recommendations` / `POST /api/admin/rollback-calibration`.
- `GET/POST /api/admin/calib/testset`, `GET /api/admin/calib/example` — тестовый набор.
- `GET /api/admin/calib/modes` — список режимов работы и текущий активный.
- `POST /api/admin/calib/run` (`{use_llm, grid, modes}`), `GET /api/admin/calib/status`,
  `POST /api/admin/calib/apply` (`{min_score,k_rerank,k_retrieve,mode}`) — автокалибровка
  параметров на тестовом наборе (по выбранным режимам; `mode` в apply переключает режим).

### Мониторинг и данные
- `GET /health` — статус, модель, бэкенд.
- `GET /api/stats` — счётчики дашборда + число чанков.
- `GET /api/logs?limit=N` — последние запросы (по умолч. 100, макс. 1000).
- `GET /api/analytics` — агрегаты для графиков.

### Настройки и режим
- `GET /api/config` — схема полей + текущие значения (секреты маскируются).
- `POST /api/config` — изменить параметры (тело — словарь ключ/значение).
- `POST /api/config/reset` — сброс к умолчаниям.
- `GET /api/mode` — текущий режим и каталог режимов.
- `POST /api/mode` — `{"mode":"basic|meta|hybrid"}`.

### Администрирование
- `GET /api/admin/status` — доступность сервисов, число чанков, статус индексации.
- `POST /api/admin/reindex` — `{"reset": bool}`.
- `POST /api/admin/apply-llm` — перезапуск vLLM с текущей моделью.
- `GET  /api/admin/web-urls` / `POST /api/admin/ingest-web` — список сайтов и их
  парсинг с добавлением в базу (`{"urls":[...]}`, до 50).
- `POST /api/admin/install-qdrant` — поднять контейнер Qdrant (фоном).
- `POST /api/admin/install-lightrag` — установить LightRAG в окружение (фоном).
- `POST /api/admin/build-graph` — установка LightRAG и построение графа (фоном).
- `POST /api/admin/finetune` — запуск пайплайна дообучения (фоном).
- `POST /api/admin/apply-finetuned` — перезапуск vLLM с LoRA-адаптером.
- `POST /api/admin/reset` — `{"targets":["index","graph","adapter","logs","settings"|"all"]}`.
- `GET  /api/admin/check-updates` — сравнение версии с GitHub.
- `POST /api/admin/update` — обновление из GitHub + перезапуск (фоном).
- `POST /api/admin/reinstall-env` — переустановка окружения (фоном).
- `POST /api/admin/reinstall-full` — `{"kind":"server"|"mac"}` полная переустановка (фоном).
- `GET  /api/admin/db/status` — статус бэкендов БД (sqlite/mysql/postgresql) и кэша Redis.
- `POST /api/admin/db/test` — `{"backend":"mysql|postgresql|sqlite"}` проверка подключения.
- `POST /api/admin/db/copy` — `{"target":...}` копировать данные и настройки в СУБД (фоном).
- `POST /api/admin/db/migrate` — `{"target":...}` копировать и переключить активную БД (фоном).
- `POST /api/admin/cache/clear` — очистить кэш Redis.
- `GET  /api/admin/catalog/status` — статус каталога в PostgreSQL и текущий источник.
- `POST /api/admin/catalog/load` — загрузить каталог документов в PostgreSQL (фоном).
- `POST /api/admin/catalog/use` — `{"source":"postgresql|filesystem"}` переключить источник каталога.
- `POST /api/admin/catalog/clear-files` — удалить из PostgreSQL только содержимое файлов.
- `POST /api/admin/restart` — перезапуск сервиса.
- `GET /api/admin/remote/hosts`, `POST /api/admin/remote/{save,delete,deploy,transfer,switch,restore}`
  — удалённые хосты: профили, деплой (qdrant/lightrag), перенос (docs/snapshot,
  push/pull), переключение бэкенда и возврат.

Пример запроса с токеном:

```bash
curl -X POST http://localhost:8000/api/mode \
  -H 'Content-Type: application/json' -H 'X-Admin-Token: ВАШ_ТОКЕН' \
  -d '{"mode":"hybrid"}'
```

## 8. Командная строка

Из корня проекта при активном `.venv`:

- `python ingest.py` — индексация (инкрементальная).
- `python ingest.py --reset` — переиндексация с нуля.
- `python -m graph_rag ingest` — построение графа знаний (hybrid+).
- `python compare.py [questions.txt|"вопрос1" "вопрос2"]` — сравнение ответов
  вектор-пайплайна и LightRAG на одинаковых вопросах.
- `python finetune/build_dataset.py` / `python finetune/train_lora.py` — сбор
  датасета и QLoRA-дообучение (или единым скриптом `bash finetune/run_pipeline.sh`).

## 9. Безопасность и эксплуатация

- **Обязательно задайте `ADMIN_TOKEN`** — иначе админка и операции доступны всем в
  сети. Менять токен можно в админке (поле «Токен администратора» в группе «Доступ»).
- Для внешнего доступа закройте сервис reverse-proxy (Caddy/Nginx) с TLS и
  авторизацией или ограничьте firewall’ом локальной сетью.
- Данные не покидают сервер: модели, индекс и документы — локальные.
- Журнал запросов хранится в `rag_logs.db` (SQLite) рядом с приложением.

Телеграм-бот

    
Доступ к базе знаний через Telegram с подтверждением пользователей
      


    
🤖 Статус бота

      

      

        
        
        
      

      
Токен бота задаётся в «Администратор» → группа «Телеграм-бот» (получить у @BotFather). После смены токена нажмите «Перезапустить бота».

    


    
🔀 Workflow бота
      

      
Путь обработки сообщения: приём → проверка доступа → тип (текст / голос → STT / файл → OCR / команда обучения) → RAG-ответ (с синонимами, прайс-папкой и внешними API) → отправка → озвучка → оценка → журнал. Неактивные ветки приглушены по текущим настройкам бота.

      

    


    
🎙 Голосовые сообщения, ответы голосом и файлы

      

      


        

          
          
Голосовое сообщение боту скачивается и распознаётся
            локальным Whisper (настройки и модель — группа «Транскрибация» в админке); распознанный
            текст обрабатывается как обычный вопрос. Рекомендация: включено. Нужны установленный бэкенд
            Whisper и ffmpeg. На слабом CPU распознавание длинных сообщений может быть медленным —
            используйте модель Whisper поменьше.

        


        

          
          
Приложенный к сообщению файл (PDF, DOCX, XLSX, изображение
            и др.) скачивается, из него извлекается текст (с OCR для картинок), и бот отвечает на подпись-вопрос
            по содержимому именно этого файла (в базу знаний не добавляется). Без подписи — присылает распознанный
            фрагмент. Рекомендация: включено; следите, чтобы были включены нужные экстракторы (OCR/CAD) и
            установлен ffmpeg.

        


        

          
          
На голосовой запрос бот, помимо текста, присылает
            озвученный ответ (синтез речи). Рекомендация: включайте, только если настроен и доступен
            TTS-движок (см. ниже). Если синтез недоступен — отправляется только текст. Длинные ответы озвучиваются
            до ~3000 символов.

        


        

          
          
Бот присылает текст ответа модели. Выключите,
            чтобы отправлять только источники и/или структуру формирования ответа без самого текста (например,
            для отладки поиска). Если выключены ответ, источники и структура — бот пришлёт короткую отметку
            об обработке. При выключенном ответе голосовой ответ не отправляется.

        


        

          
          
Под ответом — кнопки 👍 / 👎 и 💬 Комментарий.
            Оценка сохраняется сразу; по «Комментарий» бот просит написать текст, и следующее сообщение
            сохраняется как комментарий. Всё видно в журнале, на дашборде и в истории Телеграм.

        


        

          
          
Как карточки-превью в веб-чате: к ответу бот отправляет
            визуальные превью источников — миниатюры картинок, RAW-фото и чертежей, первую страницу PDF,
            кадр видео (с таймкодом) и аудио-файл. Превью строятся «лениво» и кешируются; нужны Pillow/ffmpeg.

        


        

          
          
К ответу добавляется «структура формирования ответа» —
            этапы конвейера (эмбеддинг → фильтр → поиск → BM25 → реранк → контекст → генерация, либо шаги KAG)
            с параметрами и временем. Полезно для прозрачности и отладки; выключите, чтобы бот присылал
            только ответ и источники.

        


        

          
          
          
Рекомендации: piper — лучшее качество и
            естественный русский (нужно скачать модель .onnx и указать путь ниже); say —
            хорошее качество на macOS без установки; espeak — есть везде на Linux, но «робот-голос»;
            auto подберёт доступный. Результат кодируется в OGG/Opus через ffmpeg.

        


        

          
          

            
            
          

          
          
Выберите голос из списка доступных для текущего движка или впишите вручную.
            Для espeak доступно много тембров (мужские +m1…+m7, женские +f1…+f5, шёпот);
            для say (macOS) — системные голоса (русские Milena/Yuri); для
            piper — модели .onnx из папок (DOCS_DIR/piper, ~/piper, /opt/piper). Пусто = по умолчанию.

🤖 ИИ-оценка базы знаний

      
Модель проанализирует всю проиндексированную базу
        (объём, категории, типы файлов, покрытие метаданными, длины чанков, дубликаты, примеры фрагментов и текущие
        настройки) и выдаст оценку с конкретными рекомендациями по оптимизации контента, индексации и настройкам поиска.

      

        
        
        
      

      

      

      

    


    

      
Как это работает

      

        Вы задаёте набор «вопрос → ожидаемый результат». Для каждого вопроса система
        один раз выполняет эмбеддинг → плотный поиск в Qdrant → BM25 → кросс-энкодер
        реранк по широкому пулу кандидатов. Затем перебирает сетку параметров — это лишь
        нарезка уже посчитанных кандидатов (реранк-оценки не зависят от размера пула),
        поэтому быстро и точно воспроизводит рабочий конвейер. Для каждой комбинации
        считаются метрики, выбирается лучшая. Метрики: точность — доля
        корректных (нашёл нужное / корректно промолчал); полнота — по позитивным
        вопросам; специфичность — по негативным (где правильно ответить «не знаю»).
      

    


    

      
1. Тестовый набор

      
Конструктор вопросов: для каждого задайте вопрос, ожидаемые
        файлы-источники (часть имени) и/или слова в ответе. Вопрос без
        ожиданий — негативный (правильно промолчать) — добавляйте такие, чтобы система
        не «выдумывала» на отсутствующих темах. Теги добавляются по Enter. Набор хранится в БД
        и переносится при миграции между СУБД.

      

        
        
        
        
        
        
      

      

      

        ⌨️ JSON (для опытных) — двусторонняя синхронизация с конструктором
        

          
          
        

        
      


      

        

          Сохранённые наборы вопросов (вопросы + найденные параметры)
          
          
          
          
        

        
Сохраняет текущий набор вопросов конструктора и параметры последней
          калибровки (лучшие MIN_SCORE/TOP_K по режимам). Набор можно загрузить обратно в конструктор
          или удалить.

        

      

    


    

      
Альтернатива: набор из папки test (авто Q&A)

      
Положите файлы-образцы в подпапку test каталога
        документов. По ним LLM сгенерирует пары «вопрос → эталонный ответ» (по умолчанию 50).
        При прогоне система отвечает на каждый вопрос текущим/выбранным движком, и ответ
        сравнивается с эталоном по смысловой близости (эмбеддинг-косинус). «Отклонение» =
        (1 − близость)·100%; зачёт — если отклонение ≤ заданного порога. Это оценка качества
        ответа целиком (дополняет калибровку порога/выборки выше).

      

        ✏️ Промпт генерации (LLM) — посмотреть и отредактировать
        

          
          
          
          
        

        
Промпт подаётся LLM вместе с фрагментом документа; модель должна вернуть
          JSON {"q":"…","a":"…"}. Сохранённый промпт используется при генерации и
          включается в сохранённый набор.

        
      

      

        
        
        
      

      

      


      

        

          Сохранённые наборы (промпт + пары + параметры)
          
          
          
          
        

        
Сохраняет текущий промпт, сгенерированные пары и параметры последнего
          прогона (движок, допуск, точность, MIN_SCORE/TOP_K). Набор можно загрузить обратно или удалить.

        

      


      

        
        
        
        
        
      

      
Меньше отклонение — строже зачёт (ответ должен быть ближе к эталону).
        Рекомендация: 20–35%. Прогон вызывает LLM на каждый вопрос — это занимает время.

      

      

        
График сходимости: сходство ответа с эталоном по вопросам (и накопленная доля зачётов)

        
        

          Подробный лог оценки
          
          
        

        

      

      


      

        🤖 LLM-автокалибровка (итеративная)
        
Цикл: оценка набора текущими параметрами → LLM получает структуру
          проекта, взаимодействие компонентов, их настройки, пары вопрос/ответ, тестовые файлы и
          лог предыдущей итерации → возвращает рекомендуемые параметры → они применяются → запуск
          следующей итерации. И так по кругу до лимита циклов или сходимости. Универсально для
          любого варианта работы: выберите движок и режим — оптимизатор активирует этот вариант
          и подбирает только актуальные для него параметры (порог/выборка/температура для
          вектора; режим графа для LightRAG/Hybrid+; параметры KAG для KAG; фильтры режима).
          Допуск отклонения берётся из блока оценки выше. ⚠️ Каждая итерация вызывает LLM на все
          пары + запрос параметров — это долго и затратно; начните с небольшого лимита.

        

          
          
          
          
          
          
        

        

        

          
График сходимости: точность и среднее сходство по итерациям

          
          

            Лог итераций
            
            
          

          

        

        

      

    


    

      
2. Запуск

      
Режимы работы (варианты) —
        для каких режимов подобрать параметры (порог/выборку). Режимы отличаются фильтрацией
        кандидатов перед реранком, поэтому оптимумы у них разные; применить можно по каждому
        отдельно (в т.ч. сразу переключившись на режим).

      

        
        
      

      


      
Движки (оценка end-to-end) —
        прогнать тестовый набор целиком через выбранные движки на текущих настройках и
        измерить точность. Так видно, какой вариант работы лучше на ваших данных. Параметры KAG
        и графа настраиваются вручную в их группах; здесь движки сравниваются как есть.
        ⚠️ KAG и LightRAG вызывают LLM на каждый вопрос — оценка идёт дольше.

      

        
        
      

      


      
Параметры перебора (сетка) — отметьте
        галочкой параметры компонентов, которые подбирать, и задайте диапазон и шаг.
        Снятые с галочки берутся из текущих настроек одним значением. Перебор применяется к
        калибровке режимов (для движков — оценка на текущих настройках; параметры KAG и
        индексации настраиваются отдельно — см. ниже).
        
        

      


      

        
        
        
        
      

      
Галка LLM — дополнительно генерирует ответ для рекомендованной и
        текущей комбинаций и проверяет наличие ожидаемого текста (медленнее). При выборе
        нескольких режимов калибровка идёт по очереди — время растёт пропорционально числу
        комбинаций × режимов.

Модель LLM

      

      

      
Каталог моделей (доступны к загрузке):

      

        
        
      

      

        
        
        
        
      

      
Сверху — установленные модели (выбор для генерации). Каталог — рекомендованные модели: для Ollama скачиваются, для vLLM имя подставляется в VLLM_MODEL (затем «Применить модель LLM»). Для vLLM есть отдельный выпадающий список и кнопка «Обновить список» прямо у поля VLLM_MODEL в настройках.

      

        📌 Рекомендации по выбору модели
        

          Генерация (LLM) — по железу:

          • 8–16 ГБ ОЗУ/VRAM: qwen3:8b, qwen3.6:35b-a3b-q4_K_M (MoE, быстрая), gemma3:12b.

          • 24 ГБ (RTX 3090/4090): qwen3.6:35b-a3b ✅ (лучший баланс RU/скорость), qwen2.5:32b, gemma3:27b.

          • 48–80 ГБ: qwen2.5:72b, Qwen/Qwen3.6-27B, llama3.3:70b.

          • Слабый сервер/CPU: qwen3:1.7b–qwen3:4b, llama3.2:3b.


          Для русского и RAG: Qwen3.6 / Qwen3 / Qwen2.5 — самый сильный русский; Gemma 3 — 140+ языков; Mistral-Small — точные фактологические ответы; Command-R — заточена под цитирование.

          Reasoning: Qwen3.6 и DeepSeek-R1 «думают» перед ответом — точнее на сложных вопросах, но медленнее; для простого справочного RAG берите обычные instruct-модели или просите отвечать кратко.


          Эмбеддинги (поле EMBED_MODEL в группе «Эмбеддинги и устройство», смена требует переиндексации):

          • BAAI/bge-m3 ✅ — по умолчанию, многоязычный, 1024d (dense+sparse).

          • Qwen/Qwen3-Embedding-0.6B — топ MTEB 2026, 1024d, лёгкий.

          • intfloat/multilingual-e5-large — проверенный, 1024d, хороший RU.

          • jinaai/jina-embeddings-v3 — длинные документы (1024d).

          • nomic-ai/nomic-embed-text-v2-moe — лёгкий многоязычный.

          ⚠️ При смене эмбеддера поставьте корректную EMBED_DIM (для перечисленных — 1024), затем «Сбросить индекс» и «Переиндексировать».


          Правило: для RAG качество поиска (эмбеддер + чанкинг) важнее размера LLM — хороший эмбеддер с моделью 8–14B часто лучше гигантской модели со слабым поиском.

Калибровка по оценкам ответов

      

      

        
        
        
        
      

      

      
Анализирует оценки 👍/👎 и предлагает изменения (порог релевантности, объём выборки, температура, фильтры). «Откатить» возвращает значения до последней применённой калибровки.

    


    
    

      
Обновления

      

        
        
        
      

      

      
Сравнивает версию с GitHub и обновляет код + зависимости, затем перезапускает сервис. Работает, если установка делалась из GitHub (git-репозиторий).

    


    
    

      
История чатов

      

        
        
      

      
Удаляет журнал всех запросов вместе с оценками и аналитикой (дашборд, журнал, аналитика обнулятся). Индекс, граф и настройки не затрагиваются.

    


    
    

      
Резервное копирование

      

        
        
        
        
      

      
У каждого архива считается SHA-256 и проверяется целостность. Векторный индекс Qdrant не входит в копию (он производный) — после восстановления данных выполните переиндексацию.

Переустановка и сброс

      

        
        
        
        
        
        
      

      

        
        
        
        
      

      
«Переустановить окружение» — пересоздаёт .venv и зависимости, данные сохраняются. «Полная переустановка» — DESTRUCTIVE: удаляет окружение и ВСЕ данные (индекс, граф, адаптер, журнал) и разворачивает заново; `.env` сохраняется, сервис будет недоступен во время процесса. Выбирайте кнопку под вашу платформу. На GPU нужен сервис, запущенный от пользователя с правами sudo.

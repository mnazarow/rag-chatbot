# Вариант с дообучением (fine-tuning, LoRA)

Дополнительный вариант: дообучение локальной модели на корпоративных данных
методом **QLoRA** и подача полученного адаптера через vLLM. Переключение между
базовой и дообученной моделью — из веб-панели (раздел «Администратор» →
«Модель генерации»).

> Только для GPU-варианта (Linux + NVIDIA). Обучение требует видеопамяти и времени.

## Зачем дообучение, если есть RAG

RAG и дообучение решают разные задачи и хорошо дополняют друг друга:

- **RAG** даёт знание фактов (цены, условия) с проверяемыми ссылками и мгновенным
  обновлением. Это основной механизм ответов.
- **Дообучение** меняет поведение модели: фирменный стиль и тон ответов, типовые
  формулировки, доменную терминологию, формат. Оно НЕ заменяет RAG для фактов —
  модель не должна «запоминать» меняющиеся цены.

Рекомендованный сценарий: дообученная модель **в связке с RAG** — она по-прежнему
получает контекст из документов, но отвечает в нужном стиле и лучше понимает
доменные формулировки.

## Как устроено

1. **Сбор датасета** (`build_dataset.py`): из проиндексированных чанков Qdrant
   текущая LLM генерирует пары «вопрос-ответ», строго основанные на фрагментах.
   Результат — `finetune/data/train.jsonl` в чат-формате.
2. **Обучение** (`train_lora.py`): QLoRA (4-bit) поверх базовой fp16-модели; на
   выходе — LoRA-адаптер в `finetune/adapter`. Веса не сливаются — адаптер легко
   применять и откатывать.
3. **Подача** (`docker-compose.lora.yml` + `apply_finetuned.sh`): vLLM
   перезапускается с `--enable-lora`, адаптер регистрируется как модель
   `FINETUNED_MODEL` (по умолчанию `company-lora`); базовая модель тоже остаётся
   доступной.
4. **Переключение**: настройка `USE_FINETUNED` направляет запросы на адаптер.
   Управляется карточками «Базовая / Дообученная» в админке.

## Запуск

Через веб-панель (проще):
1. «Администратор» → «Модель генерации» → **«Запустить дообучение»** (соберёт
   датасет и обучит адаптер; идёт в фоне, статус виден в «Состояние и операции»).
2. По готовности — **«Применить дообученную модель»** (перезапустит vLLM с LoRA).
3. Выбрать карточку **«Дообученная (LoRA)»**.

Через консоль:
```bash
sudo bash gpu_variant/setup_finetune.sh        # установка зависимостей + весь пайплайн
# или по шагам:
source .venv/bin/activate
python finetune/build_dataset.py               # датасет (--limit N, --per-chunk K)
python finetune/train_lora.py                  # обучение (--base, --epochs, --lr)
sudo bash gpu_variant/apply_finetuned.sh       # подать адаптер в vLLM
```

## Параметры обучения

`train_lora.py` принимает `--base` (HF id базовой fp16-модели; по умолчанию
берётся из `VLLM_MODEL` с убранным суффиксом квантизации), `--epochs` (3),
`--lr` (2e-4), `--batch`, `--grad-accum`, `--max-len`. LoRA: r=16, alpha=32.

## Требования к данным и заметки

- Качество дообучения зависит от объёма и чистоты датасета. При < ~50 парах
  `build_dataset.py` предупредит — добавьте документы или увеличьте `--per-chunk`.
- Обучайте на **fp16-базе**, подавайте адаптер на ту же базовую модель в vLLM,
  иначе адаптер несовместим.
- Адаптер в `finetune/adapter` не версионируется (см. `.gitignore`).
- Откат на базовую модель — мгновенный: карточка «Базовая» в админке.
- Дообучение не отменяет RAG: оставляйте поиск включённым, чтобы факты брались из
  документов, а стиль/терминология — от адаптера.
